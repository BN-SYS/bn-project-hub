# 1. 프로젝트 개요

## 1-1. 프로젝트 목적

멘사코리아 정회원을 대상으로 운영 가능한 전자투표 시스템을 신규 구축한다.  
기존 시스템이 쇼핑몰 회원 티켓 구매 기반으로 제한적으로 운영되는 구조를 탈피하여, **멘사코리아 정회원이라면 누구나 참여할 수 있는 독립형 전자투표 플랫폼**을 목표로 한다.

## 1-2. 기존 시스템 한계

| 한계 항목 | 내용 |
|---|---|
| 투표 자격 제한 | 쇼핑몰 회원가입 + 티켓 구매자만 직접 투표 가능 |
| 인증 방식 종속 | 주문번호(끝 5자리) 기반 인증 → 쇼핑몰 의존 |
| 투표 유형 단일화 | 총회 의결 중심의 단일 유형만 운영 가능 |
| 위임 처리 복잡 | 비구매자는 직접 접근 불가, 수임인이 대신 행사 |
| 안건 상태 단순 | 열림/닫힘 2단계만 존재, 준비중·제외 없음 |
| 모수 산정 불명확 | 안건별 실제 투표 참여자 수를 모수로 계산하여 득표율이 항상 100%로 표시됨 |
| 확장성 부재 | 총회 외 임원 선출, 헌장 개정 등 다양한 투표 운영 불가 |

## 1-3. 신규 구축 목적

- 쇼핑몰 의존 없이 **정회원 자격 기준**으로 투표권 부여
- **이니시스 간편인증** 기반의 강력한 본인확인 체계 도입
- **다양한 투표 유형**(임원 선출, 헌장 개정, 총회 의결, 커스텀)을 하나의 플랫폼에서 운영
- **구글 스프레드시트** 연동을 통한 회원 명단 관리 효율화
- **위임 관리**, **예외 처리 프로세스**, **감사 로그** 등 운영 신뢰성 확보

## 1-4. 운영 대상

- **유권자:** 관리자가 구글 스프레드시트에 등록한 기준인원 명단에 포함된 회원. 명단 마감일은 관리자(선관위 등)가 유동적으로 결정하며, 투표 진행 중에도 관리자 재량으로 추가·수정 가능. 단, 구글 스프레드시트는 시스템에서 수정을 제한하거나 감사 로그를 강제할 수 없으므로, 변경 시 시트 내 별도 컬럼 또는 메모에 사유를 기록하도록 **운영 가이드라인으로 권고**해야 한다.
- **운영자:** 멘사코리아 임원 및 지정 관리자

## 1-5. 예상 사용 시나리오

| 시나리오 | 설명 | 이번 구축 범위 |
|---|---|---|
| 임원 선출 | 후보자 개인 또는 팀 등록 후 전체 정회원 대상 선거 진행 | ✅ 포함 |
| 헌장 개정 | 개정안에 대한 찬반 투표 | ✅ 포함 |
| 긴급 안건 | 총회 외 온라인 긴급 의결 | ✅ 포함 |
| 설문/의견 수렴 | 커스텀 유형으로 운영 | ✅ 포함 |
| 정기총회 의결 | 다수 안건을 순차적으로 진행, **위임 및 정족수 관리 포함** | ❌ 이번 범위 제외 (추후 확장) |

> **정기총회 범위 제외 사유:** 정기총회는 위임 시스템(위임 신청·수임·의장 귀속 위임 처리)이 함께 적용되며, 현장 출석 체크인과 모수 산정이 복합적으로 연동되는 고도화 영역이다. 이번 구축에서는 온라인 단독 투표 방식인 임원 선출·헌장 개정·긴급 안건·설문을 우선 완성하고, 정기총회(위임 포함)는 별도 2단계로 확장 구축한다.  
> **확장 시 추가 필요 기능:** 위임 신청 플로우(U-16~U-17), 위임 관리(A-09), 출석 관리(A-10), 의장 귀속 위임 투표 행사(A-14)

# 2. 기존 시스템 분석

## 2-1. 기존 시스템 구조 요약

기존 `vote/` 폴더의 PHP 기반 시스템은 다음 구조로 운영되었다.

```
vote/
├── index.php              ← 투표자 로그인 (주문번호 인증)
├── vote.php               ← 투표 진행 화면
├── vote_action.php        ← 투표 제출 처리
├── login_action.php       ← 로그인 처리
└── manage/
    ├── index.php          ← 관리자 로그인
    ├── assemblies.php     ← 총회 목록
    ├── assembly-detail.php← 총회 상세 (안건 관리)
    ├── assembly-form.php  ← 총회 등록
    ├── agenda-form.php    ← 안건 등록
    ├── result.php         ← 결과 조회
    ├── attendance.php     ← 출석 현황
    ├── delegation.php     ← 위임 관리
    ├── bulk-vote.php      ← 의장 일괄 투표
    ├── vote-status.php    ← 투표 현황
    └── delegation_synchronization.php ← 구글 시트 동기화
```

## 2-2. 기존 인증 방식

**투표자 인증:**
1. 이름 입력
2. 전화번호 입력
3. 주문번호 끝 5자리 입력
4. 3개 조합이 구매 기록과 일치하면 로그인 허용

**전제 조건:** 쇼핑몰에서 해당 총회 티켓을 구매한 회원만 인증 가능

**관리자 인증:** ID / PW 기반 단순 로그인

## 2-3. 기존 투표 흐름

```
로그인(주문번호 인증)
→ 안건 목록 확인
→ 안건별 선택지 선택
→ 제출
→ 완료
```

- 안건은 관리자가 열림/닫힘 토글로 실시간 제어
- 위임: 구글 스프레드시트에 위임장 작성 → 관리자가 수동 동기화
- 의장 위임 투표: 의장 귀속 위임자 전원의 표를 선택지별 % 배분으로 행사

## 2-4. 기존 문제점 정리

| 문제 | 구체 내용 |
|---|---|
| 자격 제한 | 쇼핑몰 비구매 정회원은 직접 투표 불가 |
| 위임 처리 복잡성 | 비구매자 위임인은 시스템 접근 자체 불가 |
| 안건 상태 단순 | 준비중/제외 없음, 제외 안건이 "미완료"로 잘못 집계 |
| 득표율 버그 | assembly-detail.php에서 항상 100% 표시 |
| 모수 기준 불명확 | 실제 투표자 vs 총 유효 의결권 수 혼용 |
| 관리자 기능 부족 | 비밀번호 변경 화면 없음, 안건 순서 변경 삭제/재등록만 가능 |
| 무효 위임 미표시 | 무효 위임장이 화면에 표시되지 않아 민원 발생 |
| 확장성 부재 | 총회 이외 투표 유형 추가 어려운 구조 |

# 3. 신규 시스템 방향성

## 3-1. 시스템 구조 개편 방향

| 항목 | 기존 | 신규 |
|---|---|---|
| 투표 자격 | 쇼핑몰 티켓 구매자 | 멘사코리아 정회원 전체 |
| 인증 수단 | 주문번호 끝 5자리 | 이니시스 간편인증 (본인 명의 휴대폰 인증) |
| 회원 데이터 | 쇼핑몰 주문 DB | 구글 스프레드시트 (정회원 명단) |
| 투표 유형 | 총회 의결 단일 | 임원 선출 / 헌장 개정 / 총회 의결 / 커스텀 |
| 안건 상태 | 열림/닫힘 | 준비중 / 열림 / 닫힘 / 제외 |
| 위임 | 수임인(구매자)만 행사 / 스프레드시트 수동 등록 | *(추후 확장 — 정기총회 2단계)* |
| 관리자 구조 | 단일 관리자 | 단일 권한 관리자 (다중 계정 생성 가능) |
| 결과 모수 | 안건별 실제 참여자 수 (항상 100%) | **안건별** 실제 투표 참여자 수 기준 — 기준인원 대비 참여율 안건별 별도 표기 |

## 3-2. 회원 체계 방향

- **기준:** 하나의 구글 스프레드시트 내 **여러 시트** — 투표 개설 시 사용할 시트명을 지정
  - 예) "정회원" 시트 → 전체 회원 대상 투표 / "임원" 시트 → 임원 대상 투표
  - 동일 기간에 여러 투표가 열리더라도 시트가 다르면 서로 다른 기준인원 적용 가능
- **관리 항목:** 이름 / 회원번호 / 휴대폰번호 / 이메일 / 생년월일
- **회원 명단 관리 방식 (동기화 후 로컬 관리):**
  - **동기화:** 관리자가 A-08에서 "동기화" 버튼을 클릭하면 구글 스프레드시트 데이터를 시스템 DB로 1회 가져온다. 이후 회원 추가·수정·삭제는 관리자 페이지(A-08)에서 직접 처리한다.
  - **데이터 일관성 권고:** 관리자 페이지에서 회원 정보를 변경한 경우, 구글 시트에도 동일하게 반영하도록 운영 가이드라인으로 권고한다 (시스템이 자동 반영하지 않음).
  - **재동기화 경고:** 최초 동기화 이후 동기화 버튼을 다시 누르면 "구글 시트 데이터가 관리자 페이지 변경 내용을 덮어씁니다. 계속하시겠습니까?" 2단계 확인 팝업 후 진행. 반복 동기화는 불필요한 전체 데이터 읽기로 서버 부하를 유발하므로 최소화 권고.
  - **모수 확정:** **투표 전체 종료 시 단 1회** 시스템 DB 기준으로 최종 대상자 수를 확정·저장 (감사 로그 기록). DB에 존재하는 회원만 모수에 포함. 확정된 모수 기준으로 각 안건별 미참여자를 미투표로 일괄 처리. 회원번호 비교 시 하이픈/공백 제거 + 대소문자 무시 (MSK-12345 ↔ msk12345 동일 처리).
- **투표 진행 중 회원 변경 처리 정책:**

  | 케이스 | 매칭 영향 | 모수 영향 |
  |--------|----------|----------|
  | 추가 (관리자 페이지) | 즉시 인증 가능 | 투표 종료 시 DB에 있으면 포함 ✅ |
  | 이미 투표한 회원 삭제 | 이미 투표 완료 — 무관 | 투표 종료 시 DB에 없으면 **모수 제외**, vote_logs 보존(감사용). 관리자 경고 표시 ⚠️ |
  | 아직 투표 안 한 회원 삭제 | 즉시 매칭 실패 → 투표 불가 | 투표 종료 시 DB에 없으면 모수 제외 ✅ |
  | 회원 수정 (회원번호 동일) | 수정된 정보로 인증 | 회원번호 동일 시 모수 포함 ✅ |

- **기준인원 명단 마감일:** 선관위가 유동적으로 설정 가능. 투표 시작 후 관리자 페이지에서의 명단 변경은 인증 및 모수에 즉시 반영되므로 관리자 화면 내 주의 안내 필수 (아래 A-08 참조).

## 3-3. 인증 체계 방향

```
1단계: 이니시스 간편인증 (본인 명의 휴대폰 소유 증명)
  └─ 수집: 이름, 휴대폰번호, 생년월일

2단계: 회원번호 입력 (사용자 직접 입력)

3단계: 회원 매칭
  ├─ 1차: 이름 + 전화번호 + 회원번호
  └─ 2차: 이름 + 생년월일 + 회원번호 (전화번호 변경 대비)

4단계: 매칭 성공 → 투표 입장
         매칭 실패 → 예외 요청 접수
```

## 3-4. 투표 유형 확장 방향

- **템플릿 기반 설계:** 투표 유형별 템플릿을 선택하면 해당 구조로 안건 구성
- **안건 유형 조합 가능:** 하나의 투표에 찬반형/단일선택형/후보선택형 등 혼용 가능
- **향후 확장 용이:** 새 투표 유형 추가 시 템플릿만 추가하면 됨

## 3-5. 관리자 구조 방향

- **단일 권한 관리자:** 모든 관리자 계정이 동일 권한 보유 — 투표 생성/운영/결과 조회/예외 처리/계정 관리 전체 가능
- **다중 계정:** 계정은 여러 개 생성 가능, 각자 독립된 ID/PW로 로그인
- **감사 로그:** 모든 관리자 행위 기록 (누가, 언제, 무엇을)

# 4. 사용자 유형 정의

## 4-1. 사용자 유형 전체 구조

```
시스템 접근자
├── 투표자 (일반 사용자)
│   ├── 정회원 (정상 인증)
│   └── 정회원 (예외 인증 — 관리자 승인 링크)
└── 관리자 (admin) — 다중 계정 생성 가능, 모든 계정 동일 권한
```

## 4-2. 투표자 유형별 상세 정의

### 일반 정회원 (normal_member)

| 항목 | 내용 |
|---|---|
| 정의 | 이니시스 간편인증 + 회원번호 매칭 성공한 정회원 |
| 접근 범위 | 본인에게 열린 투표에만 참여 가능 |
| 투표 권한 | 직접 투표 행사 |
| 중복 투표 | 불가 (1인 1표 원칙) |

### 예외 인증 정회원 (exception_member)

| 항목 | 내용 |
|---|---|
| 정의 | 이니시스 본인인증 자체가 불가한 회원 (타인 명의 기기 사용, 해외 번호 등) → 관리자 승인 후 1회용 링크로 입장. 단순 데이터 오류(이름·전화번호 오타 등)는 구글 시트 수정으로 처리 |
| 접근 범위 | 해당 투표에만 1회 참여 가능 |
| 투표 권한 | 직접 투표 행사 |
| 로그 기록 | "관리자 예외 승인" 별도 표시 |
| 제한 | 링크 해당 투표 종료일시까지 유효, 1회 사용 후 만료 / 링크 진입 후 이름+회원번호 입력으로 본인 확인 |


## 4-3. 관리자 상세 정의

### 관리자 (admin)

| 항목 | 내용 |
|---|---|
| 정의 | 시스템 전체 기능에 접근 가능한 운영 계정 |
| 계정 관리 | 관리자 계정 생성/수정/삭제/비활성화 |
| 투표 관리 | 투표 생성/수정/삭제/강제 종료 |
| 예외 처리 | 예외 인증 요청 검토 및 처리 |
| 로그 조회 | 전체 감사 로그 열람 |
| 설정 | 시스템 전역 설정 변경 |

모든 관리자 계정은 동일한 권한을 가지며, 여러 계정을 생성하여 담당자별로 운영할 수 있다.

# 5. 사용자 투표 플로우

## 5-1. 전체 플로우 개요

사용자의 진입 경로에 따라 두 가지 플로우로 나뉘나, **모든 접근 시 회원 인증이 최우선으로 선행**되며 인증 전에는 어떠한 투표 정보도 노출되지 않는다.

### A. 포털 접속 플로우 (투표 목록 확인 후 진입)
홈페이지 공지 등을 통해 전체 투표 포털(`/evote`)로 접속하는 경우.

```
[포털 접속 (/evote)]
    ↓ (정보 노출 없이 즉시 인증 요구)
[이니시스 간편인증] ─(실패)→ 실패 안내 + 재시도
    ↓ (성공)
[회원번호 입력 및 매칭] ─(실패)→ 예외 요청 폼 (U-05)
    ↓ (성공)
[투표 목록 화면 (U-00)] ← 인증된 본인이 참여 가능한 투표만 표시
    ↓
[특정 투표 선택]
    ↓
[기투표 여부 확인] ─(이미 완료)→ 투표 완료 화면 (U-09)
    └─(미투표)→ 투표 메인 진입 (U-07)
```

### B. 개별 투표 링크 접속 플로우 (특정 투표로 바로 진입)
안내 메시지 등을 통해 특정 투표 링크(`/evote/{voteId}`)를 직접 수신하여 접속하는 경우.

```
[개별 투표 링크 접속 (/evote/{voteId})]
    ↓ (정보 노출 없이 즉시 인증 요구)
[이니시스 간편인증] ─(실패)→ 실패 안내 + 재시도
    ↓ (성공)
[회원번호 입력 및 매칭] ─(실패)→ 예외 요청 폼 (U-05)
    ↓ (성공)
[투표 접근 및 유효성 확인]
  ├─ 대상자 아님 → 접근 제한 안내 (U-12)
  ├─ 기간 종료 → 종료 안내 페이지 (U-13)
  ├─ 아직 시작 안됨 → 대기 안내 페이지 (U-14)
  └─ 진행 중 ↓
[기투표 여부 확인] ─(이미 완료)→ 투표 완료 화면 (U-09)
    └─(미투표)→ 투표 메인 진입 (U-07)
```

## 5-2. STEP별 상세 설명

### STEP 1. 투표 링크 접속

- **진입 경로:** 문자 / 알림톡 / 이메일로 발송된 URL 또는 홈페이지 공지사항
- **URL 구조:** `/evote` (포털) 또는 `/evote/{voteId}` (개별 투표)
- **우선 인증 정책:** 투표 상태(준비중/진행중/종료)나 진입 경로와 무관하게 **모든 접속자는 접속 즉시 본인인증(STEP 2) 화면으로 이동**한다. 인증을 통과하기 전까지는 투표 제목, 상태, 안건 등의 어떠한 정보도 화면에 노출되지 않는다.

### STEP 2. 이니시스 간편인증

- **진행 방식:** 이니시스 SDK 팝업 또는 임베드 형태
- **사용자 제공 정보:** 본인 명의 휴대폰으로 인증
- **수집 데이터:**

| 항목 | 설명 |
|---|---|
| authName | 인증된 이름 |
| phoneNumber | 인증된 휴대폰번호 (하이픈 제거, 01012345678 형태) |
| birthDate | 생년월일 (YYYYMMDD) |
| uniqueId | 이니시스 고유 인증 ID (중복 인증 방지용) |
| authTime | 인증 시간 |

- **오류 처리:**

| 오류 상황 | 사용자 안내 | 처리 |
|---|---|---|
| 인증 취소 | "인증을 취소하셨습니다. 다시 시도해주세요." | 재시도 버튼 표시 |
| 인증 실패 | "본인인증에 실패했습니다. 다시 시도해주세요." | 재시도 버튼 표시 |
| 통신 오류 | "통신 오류가 발생했습니다. 잠시 후 다시 시도해주세요." | 재시도 / 관리자 문의 |
| 타임아웃 (30초) | "인증 시간이 초과되었습니다." | 재시작 버튼 표시 |

### STEP 3. 회원번호 입력

- 이니시스 인증 성공 직후 노출
- 인증으로 수집된 이름을 그대로 표시하여 본인 확인 유도
- 입력 필드: 회원번호 (텍스트, 필수)
- 확인 버튼 클릭 시 서버 매칭 요청

### STEP 4. 1차 회원 매칭

**매칭 기준:** 이름 + 전화번호 + 회원번호 (3개 모두 일치)

| 정규화 항목 | 규칙 |
|---|---|
| 전화번호 | 하이픈/공백 제거 (01012345678) |
| 이름 (한글) | 양쪽 공백 제거, 정확히 일치 |
| 이름 (영문) | 소문자 통일 후 비교 |
| 회원번호 | 하이픈/공백 제거 + 대소문자 무시 후 비교 (KLM-14033 ↔ klm14033 동일 처리), 선행 0 유지 |

- **성공:** STEP 6 (중복 투표 확인)으로 이동
- **실패:** STEP 5 (2차 매칭)으로 이동

### STEP 5. 2차 회원 매칭

**매칭 기준:** 이름 + 생년월일 + 회원번호 (전화번호 변경 대비)

| 정규화 항목 | 규칙 |
|---|---|
| 생년월일 | 하이픈/슬래시 제거 후 YYYYMMDD 8자리로 통일 (1986-05-15 → 19860515) |
| 이름, 회원번호 | 1차와 동일 (회원번호 하이픈/공백 제거 + 대소문자 무시 후 비교) |

- **성공:** STEP 6으로 이동
- **실패:** STEP 5-1 (예외 요청)으로 이동

### STEP 5-1. 검증 실패 → 예외 요청 접수

**화면에 표시되는 안내:**
```
본인인증 정보가 회원 데이터와 일치하지 않습니다.
다음 중 하나의 경우일 수 있습니다:

1. 전화번호가 변경된 경우
2. 회원번호를 잘못 입력한 경우
3. 등록된 이름과 다른 경우 (개명 등)
4. 관리자 투표명단 등록 정보 오류

[ 다시 시도 ]   [ 관리자에 문의 ]
```

**예외 요청 폼 항목:**

| 항목 | 구분 | 설명 |
|---|---|---|
| 이름 | 자동 입력 (읽기전용) | 이니시스 수집값 |
| 현재 휴대폰번호 | 자동 입력 (읽기전용) | 이니시스 수집값 |
| 생년월일 | 자동 입력 (읽기전용) | 이니시스 수집값 |
| 회원번호 | 사용자 입력 | 재확인 입력 |
| 이전 휴대폰번호 | 선택 | 변경 전 번호 |
| 문제 원인 선택 | 선택 (체크박스) | 전화번호 변경 / 회원번호 오입력 / 개명 / 기타 — "기타" 선택 시 텍스트 입력 필드 표시 (직접 입력) |
| 추가 연락처 | 선택 | 관리자 답변 수신용 |

### STEP 6. 투표 접근 및 유효성 확인

인증이 완료된 사용자(세션 보유자)에 한해 접근한 투표의 정보를 확인한다.
- **대상자 아님:** 회원 명단에 없으면 "접근 권한이 없습니다." 안내 (U-12)
- **기간 종료:** "투표가 종료되었습니다." 안내 (U-13)
- **준비중:** "아직 투표가 시작되지 않았습니다." 안내 (U-14)
- **진행중:**
  - **이미 투표 완료된 경우:** 투표 화면 입장 (투표 기간 내 수정 가능, 완료 배지 표시)
  - **미투표:** 투표 화면 입장

### STEP 7. 투표 진행

- **투표 화면 진입 시 세션 생성** (투표 종료일시까지 유효, 2시간 비활성 시 간이 재확인)
- **1화면 1안건** 구조 — 한 번에 안건 1개만 표시, 이전/다음 버튼으로 순서 이동
- **상단 진행현황:** 안건 N개 중 M번째 + 완료된 안건 수 표시

**화면 레이아웃:**
```
[안건 1 / 전체 N]  ←이전  다음→

━━━━━━━━━━━━━━━━━━━━━━━
안건 제목
안건 설명
━━━━━━━━━━━━━━━━━━━━━━━

┌─────────────────────┐
│ 내 투표             │  ← 본인 카드
│ ○ 선택지A           │
│ ○ 선택지B           │
│ ○ 기권              │
│         [제출]      │
└─────────────────────┘
```

**안건별 상태 표시:**

| 상태 | 배지 색 | 배지 텍스트 | 비고 |
|---|---|---|---|
| 준비중 | 회색 | 대기중 | 선택 불가 |
| 열림 (미투표) | 파랑 | 진행중 | 선택 및 제출 가능 |
| 열림 (투표완료, 수정 가능) | 초록 | 완료 | 전체 제출 완료 전 재제출로 수정 가능 |
| 열림 (투표완료, 잠금) | 진한 초록 | 최종완료 | 전체 제출 완료 후 — 수정 불가 |
| 닫힘 (투표완료) | 회색 | 마감 | 수정 불가 |
| 닫힘 (미투표) | 옅은 빨강 | 미참여 | 미투표 자동 처리 |

### STEP 8. 투표 제출

- 안건별 "제출" 버튼 클릭 → 즉시 서버 제출
- 제출 후 해당 안건이 열림 상태이고 전체 제출 완료 전인 동안 "수정" 버튼으로 교체 → 재제출로 수정 가능
- 이미 제출된 카드는 선택값이 채워진 상태로 표시

### STEP 8-B. 전체 제출 완료 (최종 잠금)

- 투표 화면 하단에 **"전체 제출 완료"** 버튼 상시 표시
- 클릭 시 확인 팝업: "제출을 완료하면 투표 기간 내라도 모든 내용을 수정할 수 없습니다. 계속하시겠습니까?"
- 확인 후 → 해당 투표에 대해 본인의 모든 제출 내용 **영구 잠금** (서버 레벨에서 재제출 차단)
- 미제출 안건이 있는 경우 팝업에 "N개 안건이 아직 제출되지 않았습니다. 미제출 안건은 기권으로 처리됩니다." 추가 안내

### STEP 9. 투표 완료

- 완료 화면 표시: "투표에 참여해 주셔서 감사합니다."
- 투표 참여 일시 및 "전체 제출 완료" 여부 표시
- 투표 결과는 투표 종료 후 공지 예정 안내

### STEP 10. 세션 만료 처리

- 2시간 비활성으로 세션 만료 시 → 이름 + 회원번호 + 생년월일 3단계 입력 팝업 표시 (이니시스 재호출 없음)
- 입력 일치 시 세션 재개 → 기존 투표 화면으로 복귀
- 불일치 시 "정보가 일치하지 않습니다." 인라인 메시지, 재입력 가능

# 6. 인증 및 권한 정책

## 6-1. 투표자 인증 정책

### 기본 인증 수단: 이니시스 간편인증

| 항목 | 내용 |
|---|---|
| 인증 방식 | 본인 명의 휴대폰 간편인증 (PASS, 카카오, 네이버 등) |
| 인증 기관 | 이니시스 |
| 수집 데이터 | 이름, 휴대폰번호, 생년월일 |
| 인증 유효 시간 | 30초 이내 완료 요구 |
| 재인증 | 인증 실패 시 재시도 가능 (횟수 제한 없음, 단 Rate Limit 적용) |

### 회원 매칭 정책

| 매칭 단계 | 기준 | 결과 |
|---|---|---|
| 1차 매칭 | 이름 + 전화번호 + 회원번호 | 성공 시 투표 입장 |
| 2차 매칭 | 이름 + 생년월일 + 회원번호 | 성공 시 투표 입장 |
| 매칭 실패 | — | 예외 요청 접수 |

### 쇼핑몰 회원 여부와 무관한 처리

- 이니시스 인증 통과 후 구글 스프레드시트 정회원 명단과만 매칭
- 쇼핑몰 회원 여부, 티켓 구매 여부는 신규 시스템에서 무관

### 중복 로그인 정책

- 동일 회원번호로 이미 세션이 존재하는 경우: 기존 세션 무효화 후 새 세션 생성
- 로그: "동일 계정 중복 로그인 발생" 기록

### 세션 정책

> ⚠️ **이니시스 인증 비용 이슈:** 이니시스 간편인증은 호출 1회당 비용이 발생한다. 회원이 투표 기간 내 위임 수정, 재확인 등을 위해 여러 번 접속할 경우 불필요한 인증 비용이 누적될 수 있다.  
> **단기 대응:** 인증 성공 후 세션을 투표 종료일시까지 유지하여 재인증 최소화.  
> **장기 권장:** 정회원 로그인 시스템 도입 → 이니시스는 최초 1회 신원 확인 용도로만 사용, 이후 ID/PW 로그인으로 전환 (Section 16 참조).

| 항목 | 내용 |
|---|---|
| 세션 유효기간 | 투표 종료일시까지 (재인증 비용 최소화 목적) |
| 세션 저장 | 서버 사이드 세션 (PHP Session 또는 DB 기반) |
| 쿠키 설정 | Secure + HttpOnly + SameSite=Strict |
| 비활성 만료 | 2시간 비활성 시 세션 만료 → 이름 + 회원번호 + 생년월일 3단계 입력으로 세션 재개 (이니시스 재호출 없음) |

## 6-2. 관리자 인증 정책

| 항목 | 내용 |
|---|---|
| 로그인 방식 | ID + 비밀번호 |
| 비밀번호 정책 | 8자 이상, 영문+숫자+특수문자 조합 |
| 비밀번호 저장 | bcrypt 해싱 (cost factor 12 이상) |
| 세션 유효기간 | 4시간 (브라우저 닫기 시 만료) |
| 2차 인증 | 향후 도입 검토 (OTP 또는 이메일 인증) |

## 6-3. 예외 인증 정책 (이메일 인증코드)

> **적용 대상:** 이니시스 본인인증 자체가 불가한 경우에 한함 (타인 명의 기기, 해외 번호 등).  
> 단순 명단 데이터 오류(이름·전화번호·생년월일 오타 등)는 관리자가 A-08에서 수정하여 일반 인증 플로우로 처리.

**처리 흐름:**
1. 회원이 U-19(인증 불가 문의 폼)에서 이름·회원번호·이메일·사유 접수
2. 관리자가 A-13 탭 ②에서 문의 확인 → A-08 회원 명단에서 해당 회원 존재 여부 수동 확인
3. 확인 완료 시 관리자가 인증코드 발송 버튼 클릭 → 시스템이 등록된 이메일로 6자리 인증코드 자동 발송
4. 회원이 U-21(인증코드 입력 화면)에서 인증코드 입력 → 코드 일치 + 명단 내 이름·회원번호 일치 시 입장

| 항목 | 내용 |
|---|---|
| 발송 조건 | 관리자 수동 확인 후 발송 (A-08 명단에 존재하는 회원에 한함) |
| 인증코드 형식 | 6자리 숫자 (crypto.randomInt 등) |
| 유효기간 | 발송 후 24시간 또는 해당 투표 종료일시 중 먼저 도래하는 시점 |
| 사용 횟수 | 1회 (코드 입력 성공 시 즉시 만료) |
| 본인 확인 | 인증코드 입력 + 이름 + 회원번호 일치 시에만 통과 |
| 불일치 처리 | 코드 또는 정보 불일치 시 "입력 정보가 일치하지 않습니다." 재시도 가능 (5회 초과 시 잠금) |
| 회원 명단 표시 | A-08 목록에서 해당 회원 "인증코드 발송" 상태 배지 표시 |
| 로그 기록 | "관리자 예외 승인 (인증코드)" 타입으로 별도 표시, 발송 관리자 + 대상 회원번호 함께 기록 |

## 6-4. 회원 정보 불일치 처리

| 상황 | 처리 방법 |
|---|---|
| 전화번호 변경 | 2차 매칭(생년월일) 시도 → 실패 시 관리자가 A-08에서 전화번호 수정 후 재시도 안내 |
| 개명 | 이니시스 이름 ≠ 명단 이름 → 관리자가 A-08에서 이름 수정 후 재시도 안내 |
| 명단 데이터 오류 (이름·생년월일 오타 등) | 관리자가 A-08에서 직접 수정 후 회원 일반 인증 재시도 |
| 이름·생년월일 오타 수정 후 해당 회원의 예외 요청이 이미 접수된 경우 | 관리자 수동 처리 — 예외 요청 취소 또는 승인 여부 관리자가 직접 결정 |
| 회원번호 오타 | A-08에서 직접 수정. 단, 이미 해당 오타 회원번호로 투표 완료한 경우 → 투표 종료 시 모수에서 제외되고 vote_logs만 보존(감사용)되므로 투표 진행 중 수정은 각별히 주의 필요 |
| 타인 명의 기기 사용 | 이니시스 인증 자체 불가 → 관리자 이메일 인증코드 발송 |
| 해외 번호 / 이니시스 인증 불가 | 이니시스 인증 자체 불가 → 관리자 이메일 인증코드 발송 |
| 회원번호 오입력 | 재입력 안내 (재시도 버튼) |
| 중복 등록 | 관리자가 A-08에서 중복 행 정리 후 재처리 |

# 7. 투표 유형 설계

## 7-0. 진행 방식 설계

> ⚠️ **현장 투표 진행 방식 및 참석 신청 기능은 이번 구축 범위에서 제외됩니다.** 추후 정기총회(2단계) 확장 시 도입 예정입니다. 이번 단계에서는 **온라인 단독 투표** 방식으로만 운영됩니다.

투표 생성 시 **진행 방식**을 선택한다. (현시점 온라인만 제공)

| 진행 방식 | 설명 | 참석 신청 |
|---|---|---|
| 온라인 | 전체 기준인원이 온라인으로 투표 | 불필요 |
| 현장 | 현장 참석자만 투표 (총회 현장 진행) | **이번 구축 범위 제외 (추후 확장)** |

### 모수 산정 기준

| 진행 방식 | 전체 모수 구성 |
|---|---|
| 온라인 | **안건별** 실제 투표 참여자 수 (결과 분모) — 기준인원 명단 전체(대상자 수)는 **안건별** 참여율 산정 및 별도 표기 용도 |
| 현장 | 실제 출석 확정자 *(위임 포함 모수 산정은 추후 확장 — 정기총회 2단계)* |

### 참석 신청 기간 (현장 포함 시 — 추후 확장)

현장이 포함된 진행 방식에서는 투표 생성 시 참석 신청 기간을 설정한다. *(이번 구축 범위 제외)*

| 단계 | 기간 |
|---|---|
| 참석 신청 기간 | `attendance_start_at` ~ `attendance_end_at` |
| 투표 기간 | `start_at` ~ `end_at` |

- 참석 신청은 회원이 U-18 화면에서 직접 신청
- 관리자는 A-10에서 신청 목록 조회 및 현장 출석 체크인 처리
- 출석 확정 후 모수가 확정됨 (투표 시작 후 관리자가 현장 체크인 완료 처리)

## 7-1. 투표 템플릿 구조

투표는 **템플릿 + 안건 유형** 조합으로 구성된다.

```
투표 (Vote)
  └─ 템플릿 선택 (임원선출 / 헌장개정 / 총회의결 / 커스텀)
       └─ 안건 (Agenda) 1..N
            └─ 안건 유형 선택 (찬반형 / 단일선택형 / 복수선택형 / 후보선택형 / 서술형 / 참석형 / 위임형)
                 └─ 선택지 (Choice) 1..N
```

## 7-2. 안건 유형별 설계

### ① 찬반형 (binary)

| 항목 | 내용 |
|---|---|
| 선택지 | 찬성 / 반대 / 기권 (고정) |
| 사용 예 | 헌장 개정, 총회 의결, 긴급 안건 |
| 검증 | 1개만 선택 가능 |
| 결과 표시 | 찬성 N명(X%) / 반대 N명(X%) / 기권 N명(X%) / 미투표 N명(X%) |

### ② 단일선택형 (single_choice)

| 항목 | 내용 |
|---|---|
| 선택지 | 관리자가 자유 구성 (2개 이상) |
| 사용 예 | 장소 선정, 일정 투표 |
| 검증 | 1개만 선택 가능 |
| 기권 옵션 | 선택적 추가 가능 |

### ③ 복수선택형 (multi_choice)

| 항목 | 내용 |
|---|---|
| 선택지 | 관리자가 자유 구성 |
| 사용 예 | 위원 선출 (복수 선발) |
| 검증 | 관리자가 최소/최대 선택 수 지정 |
| 오류 메시지 | "최소 N개, 최대 M개 선택해주세요." |

### ④ 후보선택형 (candidate)

| 항목 | 내용 |
|---|---|
| 선택지 | 후보자 카드 (직책, 사진, 이름, 소개, 공약 포함) |
| 사용 예 | 임원 선출, 회장/부회장 선거 |
| 지원 형태 | 단독 후보 (찬반) / 복수 후보 (1인 선택) / 팀 출마 (러닝메이트) |
| 개인 후보 항목 | **직책** / 이름 / 사진 / 소개 / 공약 |
| 팀 후보 항목 | 팀명 / **팀원별 직책+이름+개인사진** / **팀 포스터(선택)** / 소개 / 공약 |
| 기권 옵션 | 항상 포함 (선거 유효성 기준) |
| 후보 상세 | 모달로 전체 소개 확인 가능 |

> ⚠️ **단일 후보 등록 시 필수 처리 정책 *(v1.2 신규)*:**  
> 후보가 1인(또는 1팀)만 등록된 경우 투표 화면은 **찬반 방식**으로 표시된다. 이 경우에도 후보자 또는 팀의 **사진, 직책, 소개, 공약** 등 후보 정보를 반드시 등록해야 하며, 미등록 시 안건 저장이 불가하다. 관리자가 후보 등록 없이 단독 찬반으로만 처리하는 것은 허용하지 않는다.

**후보 선택형 화면 차이:**

| 형태 | 등록 방식 | 투표 선택지 구성 |
|---|---|---|
| 단독 후보 (개인) | 1명 등록 — 사진·직책·소개·공약 필수 | 찬성 / 반대 / 기권 |
| 단독 후보 (팀) | 1팀 등록 — 팀원별 사진·직책 필수, 팀 소개·공약 필수 | 찬성 / 반대 / 기권 |
| 복수 후보 | 2명 이상 등록 | [직책] 후보A / [직책] 후보B / ... / 기권 |
| 팀 출마 | 2팀 이상 등록 | 팀A (직책 후보1 + 직책 후보2) / 팀B / ... / 기권 |

**후보자 카드 표시 구성 (개인):**
```
[사진]
직책: 회장
이름: 홍길동
```

**후보자 카드 표시 구성 (팀):**
```
[팀 포스터] (등록 시 표시, 미등록 시 생략)
팀명: ○○팀
  [개인사진] 회장 홍길동
  [개인사진] 사무국장 임꺽정
```

### ⑤ 서술형 (essay)

| 항목 | 내용 |
|---|---|
| 입력 | 자유 텍스트 (textarea) |
| 사용 예 | 의견 수렴, 건의사항 |
| 글자 수 | 관리자가 최대 글자 수 설정 가능 |
| 결과 | 텍스트 목록으로 표시 (통계 없음) |

### ⑥ 참석형 (attendance) — 2단계 확장 시 구현

> ⚠️ 이번 구축 범위 제외. 정기총회 2단계 확장 시 구현 예정.

## 7-3. 안건 상태 전환 규칙

```
준비중 ──→ 열림 ──⇄── 닫힘
   │          │
   ↓          ↓
  제외       제외 (닫힘 → 제외 불가)
```

| 전환 | 처리 내용 |
|---|---|
| 준비중 → 열림 | 투표 시작 |
| 열림 → 닫힘 | 미투표자 자동 미투표 처리 |
| 닫힘 → 열림 | 미투표 처리 초기화, 재투표 가능 |
| 준비중/열림 → 제외 | 2단계 확인 팝업, 기투표 데이터 내부 보존 |
| 제외 → 재포함 | 불가 (신규 등록 필요) |

## 7-4. 의결 기준 *(v1.2 신규)*

투표 결과 산정은 아래 원칙을 기본으로 한다. 투표 유형별로 적용 기준이 다르며, 결과 조회 화면(A-12)에서 해당 기준에 따른 가결/부결/당선 여부를 자동으로 표시한다.

### 기본 원칙: 다수결

| 항목 | 내용 |
|---|---|
| 적용 대상 | 임원 선출 (복수 후보), 단일선택형, 복수선택형 |
| 기준 | **실제 투표 참여자 수** 기준 가장 많은 표를 얻은 선택지가 선출/결정됨 |
| 동점 처리 | **재선거 (수동으로 신규 투표 생성하여 재진행)** |

### 찬반 의결 기준: 유효표 과반수 초과 찬성

| 항목 | 내용 |
|---|---|
| 적용 대상 | 찬반형 안건 (헌장 개정, 긴급 안건, **단독 후보 찬반** 포함) |
| 유효표 정의 | 실제 참여자 중 기권표를 제외한 찬성 + 반대 표수 |
| 가결 기준 | **찬성 수 > 유효표의 과반수 (50% 초과)** |
| 예시 | 대상자 100명, 실제 참여 70명 중 찬성 40명, 반대 20명, 기권 10명 → 유효표 60표 중 찬성 40표(66.7%) → **가결** |
| 부결 기준 | 찬성 수 ≤ 유효표의 50% 이하 |
| 비참여자 처리 | 참여율(대상자 대비) 집계에만 반영 — 결과 분모에서는 제외 |

> **정족수(쿼럼) 기준 적용:** 관리자 설정에 따라 지정된 '최소 참여율(%)' 미달 시, 해당 투표는 개표 없이 전체 무효 처리된다.

### 결과 표시 예시

**찬반형 안건 결과:**
```
대상자: 100명 | 실제 참여자: 70명 (참여율 70%)
※ 아래 % 수치는 모두 실제 참여자(70명) 기준

찬성: 40명 (57.1%)   ← 40 / 70
반대: 20명 (28.6%)   ← 20 / 70
기권: 10명 (14.3%)   ← 10 / 70  (유효표 계산 제외)
              합계: 100%

유효표(찬성+반대): 60표 | 가결 기준: 유효표의 50% 초과 = 31표
→ 찬성 40표(66.7%) → [가결] ✅
```

**단독 후보 찬반 결과:**
```
[회장] 홍길동
대상자: 100명 | 실제 참여자: 80명 (참여율 80%)
찬성: 55명 (68.8%) | 반대: 25명 (31.3%) | 기권: 0명
유효표: 80표 | 찬성 55표(68.8%) → [당선] ✅
```

## 7-5. 임원선출 표준 일정 샘플 *(v1.2 신규)*

임원선출은 아래 7단계로 진행된다. **전자투표 시스템이 관여하는 구간은 6~7번** (투표 기간 및 개표·결과 공고)이며, 1~5번은 홈페이지 공고 등 외부 채널을 통해 진행된다.

| 단계 | 일정 예시 | 처리 주체 | 시스템 관여 |
|---|---|---|---|
| 1 | 2025.01.19 | 선관위 기준일 확정 | 선관위 | ❌ |
| 2 | 2025.01.19 ~ 2025.01.30 오후 6시 | 후보 등록 기간 | 선관위 수동 처리 (시스템화 X) | ❌ |
| 3 | 2025.01.31 | 후보자 공고 및 선거운동 시작 (오전 9시부터 가능) | 선관위/홈페이지 | ❌ |
| 4 | 2025.02.03 | 등록자료 보완 마감 기한 | 선관위 | ❌ |
| 5 | 2025.02.04 오후 6시 | 선거운동 마감 | 선관위 | ❌ |
| **6** | **2025.02.05 오전 9시 ~ 2025.02.07 오후 6시** | **투표 기간 (전자투표)** | **전자투표 시스템** | ✅ |
| **7** | **2025.02.07 오후 6시 이후** | **개표 및 선거결과 공고 (당선인 즉시 임기 시작)** | **전자투표 시스템 + 선관위** | ✅ |

**시스템 운영 포인트:**
- 투표 기간(6번) 시작 전 관리자(선관위)가 후보자 정보를 시스템에 사전 등록 완료해야 함
- 투표 기간 중 추가 명부 수정이 발생할 수 있으므로, 관리자는 투표 시작 직전까지 기준인원 명단 최종 확인 필요
- 개표(7번)는 투표 종료 즉시 결과 조회 화면(A-12)에서 자동 집계 확인 가능 — 결과 공표 방식(자동/수동)은 투표 생성 시 선택
- 당선인은 결과 공표 시점부터 즉시 임기 시작 → 당선증 발급 기능(A-20) 활용 가능

| 기간 | 투표 시작일 ~ 종료일 |

### U-07. 투표 메인 (1화면 1안건)

| 항목 | 내용 |
|---|---|
| 목적 | 안건별 투표 진행 — 한 번에 1개 안건 표시, 이전/다음으로 이동 |
| 주요 컴포넌트 | 상단 진행현황 바, 안건 정보, 투표 카드(본인), 이전/다음 버튼 |
| 상태값 | 안건별 상태 (준비중/열림/닫힘) + 카드별 투표 완료 여부 |
| 사용자 액션 | 선택지 클릭, 카드별 제출, 이전/다음 이동, 후보 상세 보기 |

**화면 구성:**
```
┌─────────────────────────────────────┐
│ 안건 2 / 5   [████░░░░░] 2/5 완료  │  ← 상단 진행현황
└─────────────────────────────────────┘

  [파랑 배지: 진행중]
  제1호 안건: 헌장 개정안
  2025년 정관 제3조 개정에 관한 건...

  ┌─────────────────────────────────┐
  │ 내 투표                [완료 ✓] │  ← 이미 제출된 경우 선택값 표시
  │ ● 찬성                          │
  │ ○ 반대                          │
  │ ○ 기권                          │
  │                      [수정]     │
  └─────────────────────────────────┘

  [← 이전 안건]              [다음 안건 →]
```

**안건 상태 배지:**

| 안건 상태 | 배지 색 | 배지 텍스트 | 설명 |
|---|---|---|---|
| 열림 + 투표 완료 | 초록 | 완료 | 투표 기간 내 수정 가능 |
| 열림 + 미투표 | 파랑 | 진행중 | 지금 투표 가능 |
| 닫힘 + 투표 완료 | 회색 | 마감 | 종료됨, 투표 참여함 |
| 닫힘 + 미투표 | 옅은 빨강 | 미참여 | 종료됨, 미투표 자동 처리 |

**투표 카드 규칙:**
- 제출된 카드 (전체 제출 완료 전): 선택값 유지 + "수정" 버튼 표시 (열림 상태 안건에 한함)
- 제출된 카드 (전체 제출 완료 후): 선택값 유지 + 수정/제출 버튼 없음 (잠금 상태, "최종완료" 배지)
- 닫힘 상태 안건: 읽기전용, 제출/수정 버튼 없음

**진행현황 계산:**
```
전체(N) = 준비중 + 열림 + 닫힘 안건 수 (제외 안건 미포함)
완료(M) = 본인이 제출 완료한 안건 수 + 닫힘 상태 안건 전체
```

**안건 이동 방식 — 이전/다음 + 드롭다운:**
```
[← 이전]  [▼ 안건 2/5 헌장 개정안]  [다음 →]
              ↓ 클릭 시 드롭다운 펼침
           ┌──────────────────────────┐
           │ ✓ 1. 개회 선언    [마감] │
           │ → 2. 헌장 개정안  [진행중]│  ← 현재 안건
           │   3. 임원 선출    [진행중]│
           │   4. 예산안 심의  [대기중]│
           │   5. 폐회         [대기중]│
           └──────────────────────────┘
```
- 드롭다운 항목 클릭 → 해당 안건으로 바로 이동
- 각 항목에 안건 번호 + 제목 + 상태 배지 표시
- 첫 번째 안건에서 "이전" 버튼 비활성, 마지막 안건에서 "다음" 버튼 비활성

**모바일 고려사항:**
- 카드 세로 스크롤, 이전/다음 버튼 하단 고정 (sticky)
- 선택지 버튼 최소 44px 터치 영역
- 오프라인 대비 재시도 안내

### U-08. 후보자 소개 모달 (후보선택형 전용)

### A-04. 투표 생성/수정

**기본 정보:**
- 투표명, 투표 유형(템플릿), 기간(시작·종료 일시), 설명

**기준인원 명단 설정:**

| 설정 항목 | 설명 |
|---|---|
| 스프레드시트 ID | 시스템 전역 설정에서 관리 (투표별로 변경 불가) |
| 기준 시트명 | 해당 투표에서 사용할 시트명 입력 (예: 정회원, 임원, 이사회) |

- 시트명 입력 후 [연결 확인] 버튼 클릭 → 해당 시트 존재 여부 및 필수 컬럼(이름·회원번호·휴대폰번호) 유효성 즉시 검증
- 동일 기간 복수 투표가 열려도 시트명이 다르면 각각 독립된 기준인원 적용
- 저장 후 별도 스냅샷 생성 없음 — 인증은 항상 구글 시트 실시간 조회 (서버 캐시 5~10분)
- 투표 전체 종료 시 단 1회 시트 조회 → 전체 모수 자동 확정·저장 (감사 로그 기록)

**진행 방식 설정:**

| 설정 항목 | 옵션 | 기본값 |
|---|---|---|
| 진행 방식 | 온라인 / 현장 | 온라인 (현장은 이번 구축 범위 제외) |

- 현장 선택 시 참석 신청 기간 설정 영역 표시 *(현장 방식 추후 확장 전까지는 비활성화)*

**참석 신청 기간 설정 (현장 포함 시만 표시 — 추후 확장):**

| 설정 항목 | 설명 |
|---|---|
| 참석 신청 시작 일시 | 필수 입력 |
| 참석 신청 종료 일시 | 투표 시작 일시 이전이어야 함 |

- 참석 신청 종료 일시가 투표 시작 일시보다 늦으면 저장 불가 + 인라인 오류 표시

**운영 설정 (Kevoting 벤치마킹 반영):**

| 설정 항목 | 옵션 | 기본값 |
|---|---|---|
| 시작 방식 | 자동(예약 시작) / 수동(관리자 직접 시작) | 자동 |
| 결과 공표 방식 | **관리자 수동 공표** (단일 고정) | 수동 공표 |
| 투표확인증 발급 | 활성 / 비활성 | **이번 구축 범위 제외 (추후 확장)** |
| 확인증 발급 시점 | 투표 완료 즉시 / 투표 종료 후 | **이번 구축 범위 제외 (추후 확장)** |
| 기명/무기명 | **기명(투표자 식별 가능)** (단일 고정, 무기명 미지원) | 기명 |
| 기권 처리 방식 | 안건별 개별 기권 버튼 / 투표 시작 시 일괄 기권 선언 | 개별 기권 |
| 당선증 발급 | 활성 / 비활성 (후보선택형 안건 존재 시만 표시) | 비활성 |

### A-08. 회원 명단 관리

**기능:**
- **구글 스프레드시트 동기화 버튼:** 시트 데이터를 시스템 DB로 1회 가져오기
  - 최초 동기화: 즉시 실행
  - 재동기화: "현재 관리자 페이지 변경 내용이 구글 시트 데이터로 덮어써집니다. 계속하시겠습니까?" 2단계 확인 필수
- 동기화 시점 표시 (마지막 동기화: YYYY-MM-DD HH:MM)
- 회원 목록 조회 (이름 / 회원번호 / 전화번호 / 이메일 / 생년월일 / 인증코드 발송 여부)
- 검색 및 정렬 (이름, 회원번호)
- CSV 다운로드 (전체 명단)
- 개별 회원 추가 (관리자 직접 입력)
- 개별 회원 수정 (투표 진행 중에도 허용 — 아래 삭제 정책 주의)
- 개별 회원 삭제 (투표 진행 중에도 허용 — 단, 아래 삭제 정책 준수 필수)
- **데이터 일관성 안내 배너:** "관리자 페이지에서 변경한 내용은 구글 시트에 자동 반영되지 않습니다. 변경 후 구글 시트에도 동일하게 반영해 주세요."

**기준인원 명단 마감일 유동 적용 *(v1.2 신규)*:**  
기준인원 명단의 마감일은 선관위가 유동적으로 결정하며, 투표 진행 중에도 관리자 페이지(A-08)에서 명단을 추가·수정·삭제할 수 있다. 단, 아래를 **운영 가이드라인**으로 준수할 것을 권고한다:

- 투표 진행 중 명단 변경은 선관위 협의 후 처리
- 관리자 페이지에서 변경한 내용은 구글 시트에 자동 반영되지 않으므로, 변경 후 구글 시트에도 동일하게 수동 반영 필요
- 동기화 버튼은 구글 시트 내용이 관리자 페이지 데이터를 덮어쓰므로, 관리자 페이지 변경 내용이 있는 경우 동기화 전 반드시 확인

**명단 삭제 정책 *(v1.2 신규)*:**

> ⚠️ **삭제는 가급적 하지 않도록 강력히 권고합니다.**  
> 정보 오류 정정이 목적인 경우, 삭제 후 재등록보다 **수정(이름·전화번호·생년월일 등 교정)을 우선 사용**해주세요.

| 상황 | 처리 결과 |
|---|---|
| 아직 투표하지 않은 회원 삭제 | 이후 모수 및 투표 참여에서 제외 |
| **이미 투표를 완료한 회원 삭제** | **투표 결과(vote_logs)에는 영향 없음 — 기 투표 데이터는 영구 보존** |
| 삭제된 회원의 기 투표 데이터 | 민원 발생 시 vote_logs 및 구글 시트 변경 이력(파일 → 변경 이력 보기)으로 확인 가능 |

삭제 처리 시 UI에 아래 안내 문구를 필수 표시한다:
```

**명부 업로드 실패 목록 *(Kevoting 벤치마킹)*:**

CSV 업로드 또는 Google Sheets 동기화 시 형식 오류가 있는 행은 업로드를 중단하지 않고 **정상 행만 처리 후 실패 목록을 별도 표시**한다.

|---|---|

**처리 흐름:**
1. 업로드 완료 후 실패 행이 있으면 경고 배너 표시: "N개 행이 등록되지 않았습니다."
2. "실패 목록 보기" 버튼 → 실패 행 테이블 펼침
3. 관리자가 원본 파일을 수정 후 재업로드 또는 개별 수동 추가
4. 실패 행은 모수 계산 전 반드시 확인 필요

# 12. 보안 고려사항

## 12-1. 인증 및 접근 제어

| 항목 | 적용 방안 |
|---|---|
| 중복 투표 방지 | vote_logs 테이블에 member_number + vote_id + agenda_id UNIQUE 제약 |
| 이중 인증 | 이니시스 인증 + 회원번호 매칭 2단계 |
| Rate Limiting | 동일 IP 기준 분당 10회 이상 인증 요청 차단 |
| 예외 요청 어뷰징 방지 | 동일 투표 + 동일 이름 + 동일 회원번호 중복 요청 차단 |
| 세션 고정 공격 방지 | 로그인 후 세션 ID 재생성 (session_regenerate_id) |

## 12-2. 데이터 보호

| 항목 | 적용 방안 |
|---|---|
| 민감 정보 암호화 | 생년월일, 휴대폰번호 → AES-256 암호화 저장 |
| 비밀번호 해싱 | bcrypt (cost factor 12 이상) |
| HTTPS 강제 | HTTP → HTTPS 리다이렉트, HSTS 적용 |
| 쿠키 보안 | Secure + HttpOnly + SameSite=Strict |
| SQL Injection | PDO Prepared Statement 사용 |
| XSS 방지 | 출력 시 htmlspecialchars 처리, CSP 헤더 |
| CSRF 방지 | 폼 제출 시 CSRF 토큰 검증 |

## 12-3. 투표 무결성

| 항목 | 적용 방안 |
|---|---|
| 결과 조작 방지 | vote_logs 테이블 직접 수정 불가 (관리자 포함) |
| 감사 로그 | 모든 투표 행위, 관리자 행위 기록 (삭제 불가) |
| IP 기록 | 모든 인증 및 투표 시 IP 기록 |
| 이니시스 uniqueId | 동일 인증 ID로 중복 투표 방지 |
| 위임 무결성 | 위임 신청 후 본인 직접 투표 불가 처리 (DB 레벨) |
| 수임인 검증 최소 응답 | 이름+회원번호 일치 여부(bool)만 반환 — 회원 정보 조회 불가, 자동완성·검색 미제공 |
| 회원 명단 수정 감사 추적 | 관리자 페이지(A-08)에서의 추가·수정·삭제는 감사 로그에 기록 (수정자·수정 일시 포함). 모수 확정 시점(투표 종료 시)의 최종 명단 상태도 시스템 로그에 기록. 구글 시트 변경은 구글 시트 자체 변경 이력(파일 → 변경 이력 보기) 활용 권고 |

## 12-4. 인프라 보안

| 항목 | 적용 방안 |
|---|---|
| 관리자 접근 제한 | `/manage/` 경로 IP 화이트리스트 고려 |
| 로그 파일 노출 방지 | 로그 파일 웹 루트 외부 저장 |
| 에러 메시지 | 운영 환경에서 상세 에러 미노출 (로그로만 기록) |
| 파일 업로드 | MIME 타입 + 확장자 이중 검증, 업로드 경로 웹 루트 외부 |
| 이니시스 콜백 검증 | 이니시스 서명 키 검증 필수 |
