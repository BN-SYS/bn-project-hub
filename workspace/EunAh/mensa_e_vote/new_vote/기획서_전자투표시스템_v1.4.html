1프로젝트 개요

1-1. 프로젝트 목적

멘사코리아 정회원을 대상으로 운영 가능한 전자투표 시스템을 신규 구축한다. 기존 시스템이 쇼핑몰 회원 티켓 구매 기반으로 제한적으로 운영되는 구조를 탈피하여, 멘사코리아 정회원이라면 누구나 참여할 수 있는 독립형 전자투표 플랫폼을 목표로 한다.

1-2. 기존 시스템 한계

한계 항목내용
투표 자격 제한쇼핑몰 회원가입 + 티켓 구매자만 직접 투표 가능
인증 방식 종속주문번호(끝 5자리) 기반 인증 → 쇼핑몰 의존
투표 유형 단일화총회 의결 중심의 단일 유형만 운영 가능
안건 상태 단순열림/닫힘 2단계만 존재, 준비중·제외 없음
모수 산정 불명확안건별 실제 투표 참여자 수를 모수로 계산 → 득표율 항상 100%
확장성 부재총회 외 임원 선출, 헌장 개정 등 다양한 투표 운영 불가

1-3. 신규 구축 목적

  • 쇼핑몰 의존 없이 정회원 자격 기준으로 투표권 부여
  • 이니시스 간편인증 기반의 강력한 본인확인 체계 도입
  • 다양한 투표 유형(임원 선출, 헌장 개정, 총회 의결, 커스텀)을 하나의 플랫폼에서 운영
  • 구글 스프레드시트 연동을 통한 회원 명단 관리 효율화
  • 감사 로그, 예외 처리 프로세스 등 운영 신뢰성 확보

1-4. 운영 대상

유권자: 관리자가 구글 스프레드시트에 등록한 기준인원 명단에 포함된 회원. 명단 마감일은 선관위가 유동적으로 설정. 운영자: 멘사코리아 임원 및 지정 관리자.

1-5. 예상 사용 시나리오

시나리오설명이번 범위
임원 선출후보자 등록 후 전체 정회원 대상 선거✅ 포함
헌장 개정개정안에 대한 찬반 투표✅ 포함
긴급 안건총회 외 온라인 긴급 의결✅ 포함
설문/의견 수렴커스텀 유형으로 운영✅ 포함
정기총회 의결위임 및 정족수 관리 포함❌ 2단계 확장
정기총회 범위 제외 사유 위임 시스템(신청·수임·의장 귀속 위임)과 현장 출석 체크인이 복합 연동되는 고도화 영역. 이번 단계에서는 온라인 단독 투표를 우선 완성하고 정기총회(위임 포함)는 2단계로 확장.

2기존 시스템 분석

2-1. 기존 시스템 구조

vote/
├── index.php              ← 투표자 로그인 (주문번호 인증)
├── vote.php               ← 투표 진행 화면
├── vote_action.php        ← 투표 제출 처리
└── manage/
    ├── assemblies.php     ← 총회 목록
    ├── assembly-detail.php← 총회 상세 (안건 관리)
    ├── result.php         ← 결과 조회
    ├── attendance.php     ← 출석 현황
    ├── delegation.php     ← 위임 관리
    └── delegation_synchronization.php ← 구글 시트 동기화

2-2. 기존 인증 방식

이름 + 전화번호 + 주문번호 끝 5자리 → 3개 조합이 구매 기록과 일치하면 로그인. 쇼핑몰 티켓 구매자만 인증 가능.

2-4. 기존 문제점

문제구체 내용
자격 제한쇼핑몰 비구매 정회원은 직접 투표 불가
안건 상태 단순준비중/제외 없음, 제외 안건이 "미완료"로 잘못 집계
득표율 버그assembly-detail.php에서 항상 100% 표시
모수 기준 불명확실제 투표자 vs 총 유효 의결권 수 혼용
관리자 기능 부족비밀번호 변경 화면 없음, 안건 순서 변경 불가

3신규 시스템 방향성

3-1. 시스템 구조 개편

항목기존신규
투표 자격쇼핑몰 티켓 구매자멘사코리아 정회원 전체
인증 수단주문번호 끝 5자리이니시스 간편인증 (본인 명의 휴대폰)
회원 데이터쇼핑몰 주문 DB구글 스프레드시트 (정회원 명단)
투표 유형총회 의결 단일임원 선출 / 헌장 개정 / 총회 의결 / 커스텀
안건 상태열림/닫힘준비중 / 열림 / 닫힘 / 제외
결과 모수안건별 실제 참여자 수 (항상 100%)안건별 실제 참여자 수 + 기준인원 대비 참여율 별도 표기

3-2. 회원 체계 — 하이브리드 방식

실시간 매칭 + 투표 종료 시 모수 확정 인증 시 구글 시트 실시간 조회(서버 캐시 5~10분). DB 스냅샷 불필요.
모수 확정: 투표 전체 종료 시 단 1회 시트 조회 → 최종 대상자 수 확정·저장 → 각 안건별 미참여자를 미투표로 일괄 처리.
케이스매칭 영향모수 영향
회원 추가캐시 갱신 후 즉시 인증 가능투표 종료 시 시트에 있으면 포함 ✅
이미 투표한 회원 삭제무관 (완료)시트에 없으면 모수 제외, vote_logs 보존 ⚠️
아직 투표 안 한 회원 삭제캐시 갱신 후 매칭 실패시트에 없으면 모수 제외 ✅
회원 수정 (회원번호 동일)갱신 후 수정된 정보로 인증회원번호 동일 시 모수 포함 ✅

6인증 및 권한 정책

6-1. 세션 정책

⚠️ 이니시스 인증 비용 이슈 간편인증은 호출 1회당 비용 발생. 세션을 투표 종료일시까지 유지하여 재인증 최소화.
장기 권장: 정회원 로그인 시스템 도입 → 이니시스는 최초 1회 신원 확인 용도로만 사용 (Phase 4).
항목내용
세션 유효기간투표 종료일시까지
비활성 만료2시간 비활성 시 세션 만료 → 이름+회원번호+생년월일 3단계 입력으로 세션 재개
쿠키 설정Secure + HttpOnly + SameSite=Strict
중복 로그인기존 세션 무효화 후 새 세션 생성

6-3. 예외 인증 정책 (1회용 링크)

적용 대상이니시스 본인인증 자체가 불가한 경우에 한함 (타인 명의 기기, 해외 번호 등). 단순 명단 데이터 오류는 관리자가 구글 시트 직접 수정으로 처리.
항목내용
발급 조건관리자 승인 (회원 명단에서 대상 회원 직접 선택 후 발급)
링크 유효기간해당 투표 종료일시까지
사용 횟수1회 (이름+회원번호 확인 완료 및 세션 생성 시 즉시 만료)
토큰 길이64자 (crypto.randomBytes(32).toString('hex'))
로그"관리자 예외 승인" 타입으로 별도 표시

7투표 유형 설계

투표 구조 투표(Vote) → 템플릿 선택(임원선출/헌장개정/총회의결/커스텀) → 안건(Agenda) 1..N → 안건 유형 선택 → 선택지(Choice) 1..N
