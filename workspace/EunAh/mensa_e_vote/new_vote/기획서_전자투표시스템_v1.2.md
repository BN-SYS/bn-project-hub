# 1. 프로젝트 개요

## 1-1. 프로젝트 목적

멘사코리아 정회원을 대상으로 운영 가능한 전자투표 시스템을 신규 구축한다.  
기존 시스템이 쇼핑몰 기반으로 제한적으로 운영되어 왔던 구조를 탈피하여,  
**멘사코리아 정회원이라면 누구나 참여할 수 있는 독립형 전자투표 플랫폼**을 목표로 한다.

## 1-2. 기존 시스템 한계

| 한계 항목 | 내용 |
|---|---|
| 투표 자격 제한 | 쇼핑몰 회원가입 + 티켓 구매자만 직접 투표 가능 |
| 인증 방식 종속 | 주문번호(끝 5자리) 기반 인증 → 쇼핑몰 의존 |
| 투표 유형 단일화 | 총회 의결 중심의 단일 유형만 운영 가능 |
| 위임 처리 복잡 | 비구매자는 직접 접근 불가, 수임인이 대신 행사 |
| 안건 상태 단순 | 열림/닫힘 2단계만 존재, 준비중·제외 없음 |
| 모수 산정 불명확 | 안건별 실제 투표 참여자 수를 모수로 계산하여 득표율이 항상 100%로 표시됨 |
| 확장성 부재 | 총회 외 임원 선출, 헌장 개정 등 다양한 투표 운영 불가 |

## 1-3. 신규 구축 목적

- 쇼핑몰 의존 없이 **정회원 자격 기준**으로 투표권 부여
- **이니시스 간편인증** 기반의 강력한 본인확인 체계 도입
- **다양한 투표 유형**(임원 선출, 헌장 개정, 총회 의결, 커스텀)을 하나의 플랫폼에서 운영
- **구글 스프레드시트** 연동을 통한 회원 명단 관리 효율화
- **위임 관리**, **예외 처리 프로세스**, **감사 로그** 등 운영 신뢰성 확보

## 1-4. 운영 대상

- **유권자:** 관리자가 구글 스프레드시트에 등록한 기준인원 명단에 포함된 회원. 명단 마감일은 관리자(선관위 등)가 유동적으로 결정하며, 투표 진행 중에도 관리자 재량으로 추가·수정 가능 (변경 시 2단계 확인 + 사유 입력 + 감사 로그 기록 필수)
- **운영자:** 멘사코리아 임원 및 지정 관리자

## 1-5. 예상 사용 시나리오

| 시나리오 | 설명 | 이번 구축 범위 |
|---|---|---|
| 임원 선출 | 후보자 개인 또는 팀 등록 후 전체 정회원 대상 선거 진행 | ✅ 포함 |
| 헌장 개정 | 개정안에 대한 찬반 투표 | ✅ 포함 |
| 긴급 안건 | 총회 외 온라인 긴급 의결 | ✅ 포함 |
| 설문/의견 수렴 | 커스텀 유형으로 운영 | ✅ 포함 |
| 정기총회 의결 | 다수 안건을 순차적으로 진행, **위임 및 정족수 관리 포함** | ❌ 이번 범위 제외 (추후 확장) |

> **정기총회 범위 제외 사유:** 정기총회는 위임 시스템(위임 신청·수임·의장 귀속 위임 처리)이 함께 적용되며, 현장 출석 체크인과 모수 산정이 복합적으로 연동되는 고도화 영역이다. 이번 구축에서는 온라인 단독 투표 방식인 임원 선출·헌장 개정·긴급 안건·설문을 우선 완성하고, 정기총회(위임 포함)는 별도 2단계로 확장 구축한다.  
> **확장 시 추가 필요 기능:** 위임 신청 플로우(U-16~U-17), 위임 관리(A-09), 출석 관리(A-10), 의장 귀속 위임 투표 행사(A-14)

# 2. 기존 시스템 분석

## 2-1. 기존 시스템 구조 요약

기존 `vote/` 폴더의 PHP 기반 시스템은 다음 구조로 운영되었다.

```
vote/
├── index.php              ← 투표자 로그인 (주문번호 인증)
├── vote.php               ← 투표 진행 화면
├── vote_action.php        ← 투표 제출 처리
├── login_action.php       ← 로그인 처리
└── manage/
    ├── index.php          ← 관리자 로그인
    ├── assemblies.php     ← 총회 목록
    ├── assembly-detail.php← 총회 상세 (안건 관리)
    ├── assembly-form.php  ← 총회 등록
    ├── agenda-form.php    ← 안건 등록
    ├── result.php         ← 결과 조회
    ├── attendance.php     ← 출석 현황
    ├── delegation.php     ← 위임 관리
    ├── bulk-vote.php      ← 의장 일괄 투표
    ├── vote-status.php    ← 투표 현황
    └── delegation_synchronization.php ← 구글 시트 동기화
```

## 2-2. 기존 인증 방식

**투표자 인증:**
1. 이름 입력
2. 전화번호 입력
3. 주문번호 끝 5자리 입력
4. 3개 조합이 구매 기록과 일치하면 로그인 허용

**전제 조건:** 쇼핑몰에서 해당 총회 티켓을 구매한 회원만 인증 가능

**관리자 인증:** ID / PW 기반 단순 로그인

## 2-3. 기존 투표 흐름

```
로그인(주문번호 인증)
→ 안건 목록 확인
→ 안건별 선택지 선택
→ 제출
→ 완료
```

- 안건은 관리자가 열림/닫힘 토글로 실시간 제어
- 위임: 구글 스프레드시트에 위임장 작성 → 관리자가 수동 동기화
- 의장 위임 투표: 의장 귀속 위임자 전원의 표를 선택지별 % 배분으로 행사

## 2-4. 기존 문제점 정리

| 문제 | 구체 내용 |
|---|---|
| 자격 제한 | 쇼핑몰 비구매 정회원은 직접 투표 불가 |
| 위임 처리 복잡성 | 비구매자 위임인은 시스템 접근 자체 불가 |
| 안건 상태 단순 | 준비중/제외 없음, 제외 안건이 "미완료"로 잘못 집계 |
| 득표율 버그 | assembly-detail.php에서 항상 100% 표시 |
| 모수 기준 불명확 | 실제 투표자 vs 총 유효 의결권 수 혼용 |
| 관리자 기능 부족 | 비밀번호 변경 화면 없음, 안건 순서 변경 삭제/재등록만 가능 |
| 무효 위임 미표시 | 무효 위임장이 화면에 표시되지 않아 민원 발생 |
| 확장성 부재 | 총회 이외 투표 유형 추가 어려운 구조 |

# 3. 신규 시스템 방향성

## 3-1. 시스템 구조 개편 방향

| 항목 | 기존 | 신규 |
|---|---|---|
| 투표 자격 | 쇼핑몰 티켓 구매자 | 멘사코리아 정회원 전체 |
| 인증 수단 | 주문번호 끝 5자리 | 이니시스 간편인증 (본인 명의 휴대폰 인증) |
| 회원 데이터 | 쇼핑몰 주문 DB | 구글 스프레드시트 (정회원 명단) |
| 투표 유형 | 총회 의결 단일 | 임원 선출 / 헌장 개정 / 총회 의결 / 커스텀 |
| 안건 상태 | 열림/닫힘 | 준비중 / 열림 / 닫힘 / 제외 |
| 위임 | 수임인(구매자)만 행사 / 스프레드시트 수동 등록 | 위임인이 시스템에서 직접 신청 (별도 위임 신청 기간 운영) |
| 관리자 구조 | 단일 관리자 | 단일 권한 관리자 (다중 계정 생성 가능) |
| 결과 모수 | 안건별 실제 참여자 수 (항상 100%) | 실제 투표 참여자 수 기준 — 기준인원 명단 전체(대상자 수)는 별도 표시 |

## 3-2. 회원 체계 방향

- **기준:** 하나의 구글 스프레드시트 내 **여러 시트** — 투표 개설 시 사용할 시트명을 지정
  - 예) "정회원" 시트 → 전체 회원 대상 투표 / "임원" 시트 → 임원 대상 투표
  - 동일 기간에 여러 투표가 열리더라도 시트가 다르면 서로 다른 기준인원 적용 가능
- **관리 항목:** 이름 / 회원번호 / 휴대폰번호 / 이메일 / 생년월일
- **혼합 방식 (스냅샷 + 재동기화):** 투표 생성 시 지정한 시트의 명단을 DB에 스냅샷으로 저장. 이후 관리자가 [구글 시트 재동기화] 버튼으로 시트의 최신 명단을 다시 불러와 스냅샷을 갱신할 수 있다. 재동기화 시 2단계 확인 + 사유 입력 + 감사 로그 기록 필수.
- **투표 진행 중 재동기화 정책:** 투표 상태가 `열림`일 때 재동기화를 실행하면 **추가·수정만 반영**되며, 시트에서 제거된 회원은 스냅샷에서 삭제하지 않는다 (이미 부여된 투표권 유지). `준비중` / `닫힘` 상태에서는 전체 갱신(추가·수정·삭제) 허용.
- **기준인원 명단 마감일 유동 적용:** 명단 마감일은 선관위가 유동적으로 설정할 수 있으며, 투표 진행 중에도 관리자가 기준인원 명단을 추가하거나 수정할 수 있다. 단, 모든 변경은 **2단계 확인 + 수정 사유 필수 입력 + 감사 로그 기록** 절차를 따른다.
- **명단 삭제 정책:** 삭제는 가급적 지양한다. 오류가 있는 경우 삭제보다 수정(이름·전화번호·생년월일 등 정보 교정)을 우선 권장한다. 투표 진행 중에는 재동기화를 통한 삭제 반영이 차단되며, 삭제가 반드시 필요한 경우에는 처리 사유를 반드시 기록하도록 안내 문구를 UI에 명시한다. 기투표 데이터는 vote_logs에 영구 보존된다.

## 3-3. 인증 체계 방향

```
1단계: 이니시스 간편인증 (본인 명의 휴대폰 소유 증명)
  └─ 수집: 이름, 휴대폰번호, 생년월일

2단계: 회원번호 입력 (사용자 직접 입력)

3단계: 회원 매칭
  ├─ 1차: 이름 + 전화번호 + 회원번호
  └─ 2차: 이름 + 생년월일 + 회원번호 (전화번호 변경 대비)

4단계: 매칭 성공 → 투표 입장
         매칭 실패 → 예외 요청 접수
```

## 3-4. 투표 유형 확장 방향

- **템플릿 기반 설계:** 투표 유형별 템플릿을 선택하면 해당 구조로 안건 구성
- **안건 유형 조합 가능:** 하나의 투표에 찬반형/단일선택형/후보선택형 등 혼용 가능
- **향후 확장 용이:** 새 투표 유형 추가 시 템플릿만 추가하면 됨

## 3-5. 관리자 구조 방향

- **단일 권한 관리자:** 모든 관리자 계정이 동일 권한 보유 — 투표 생성/운영/결과 조회/예외 처리/계정 관리 전체 가능
- **다중 계정:** 계정은 여러 개 생성 가능, 각자 독립된 ID/PW로 로그인
- **감사 로그:** 모든 관리자 행위 기록 (누가, 언제, 무엇을)

# 4. 사용자 유형 정의

## 4-1. 사용자 유형 전체 구조

```
시스템 접근자
├── 투표자 (일반 사용자)
│   ├── 정회원 (정상 인증)
│   ├── 정회원 (예외 인증 — 관리자 승인 링크)
│   └── 위임자 (위임 후 본인 투표 취소된 경우)
└── 관리자 (admin) — 다중 계정 생성 가능, 모든 계정 동일 권한
```

## 4-2. 투표자 유형별 상세 정의

### 일반 정회원 (normal_member)

| 항목 | 내용 |
|---|---|
| 정의 | 이니시스 간편인증 + 회원번호 매칭 성공한 정회원 |
| 접근 범위 | 본인에게 열린 투표에만 참여 가능 |
| 투표 권한 | 직접 투표 행사 |
| 위임 가능 | 타 정회원에게 위임 또는 의장 위임 |
| 중복 투표 | 불가 (1인 1표 원칙) |

### 예외 인증 정회원 (exception_member)

| 항목 | 내용 |
|---|---|
| 정의 | 이니시스 본인인증 자체가 불가한 회원 (타인 명의 기기 사용, 해외 번호 등) → 관리자 승인 후 1회용 링크로 입장. 단순 데이터 오류(이름·전화번호 오타 등)는 스냅샷 수정으로 처리 |
| 접근 범위 | 해당 투표에만 1회 참여 가능 |
| 투표 권한 | 직접 투표 행사 |
| 로그 기록 | "관리자 예외 승인" 별도 표시 |
| 제한 | 링크 해당 투표 종료일시까지 유효, 1회 사용 후 만료 / 링크 진입 후 이름+회원번호 입력으로 본인 확인 |

### 위임인 (delegator)

| 항목 | 내용 |
|---|---|
| 정의 | 타 회원 또는 의장에게 투표를 위임한 회원 |
| 투표 권한 | 위임 후 본인 직접 투표 불가 |
| 예외 | 현장 포함 투표인 경우에 한해, 수임인 투표 전 현장 출석(검표) 시 관리자가 위임 취소 처리 → 직접 투표 복귀. 순수 온라인 투표는 위임 신청 기간 내 본인 직접 취소만 가능 |
| 수임인 불참 처리 | 현장 포함 투표에서 수임인이 현장 미출석 시 해당 위임 건 무효 처리 → 위임인·수임인 모두 모수에서 제외 (의장 귀속 위임은 의장이 항상 출석하므로 해당 없음) |
| 모수 포함 | 기준인원 명단에 있는 경우 포함 |

### 수임인 (delegatee)

| 항목 | 내용 |
|---|---|
| 정의 | 타 회원의 위임을 받은 정회원 |
| 투표 권한 | 본인 투표 + 위임받은 건 추가 행사 |
| 전제 | 본인도 기준인원 명단에 포함되어야 유효 위임 |
| 의장 위임 | 관리자(의장)가 시스템에서 일괄 행사 |

## 4-3. 관리자 상세 정의

### 관리자 (admin)

| 항목 | 내용 |
|---|---|
| 정의 | 시스템 전체 기능에 접근 가능한 운영 계정 |
| 계정 관리 | 관리자 계정 생성/수정/삭제/비활성화 |
| 투표 관리 | 투표 생성/수정/삭제/강제 종료 |
| 예외 처리 | 예외 인증 요청 검토 및 처리 |
| 로그 조회 | 전체 감사 로그 열람 |
| 설정 | 시스템 전역 설정 변경 |

모든 관리자 계정은 동일한 권한을 가지며, 여러 계정을 생성하여 담당자별로 운영할 수 있다.

# 5. 사용자 투표 플로우

## 5-0. 위임 신청 플로우 (투표 시작 전 별도 기간)

위임은 투표 단위로 처리하며, **투표 시작 전 별도의 위임 신청 기간**에 접수한다.  
위임 신청 기간이 설정된 투표에 한해서만 해당 플로우가 활성화된다.

```
[위임 신청 링크 접속] (/vote/{voteId}/delegate)
    ↓
[위임 신청 기간 확인]
  ├─ 기간 전 → "위임 신청 기간이 아직 시작되지 않았습니다." 안내
  ├─ 기간 종료 → "위임 신청이 마감되었습니다." 안내
  └─ 기간 중 → 이니시스 간편인증으로 이동
    ↓
[이니시스 간편인증]
    ↓
[회원번호 입력 + 회원 매칭]
  ├─ 매칭 실패 → 예외 요청 폼 (투표 예외 요청과 동일 프로세스)
  └─ 매칭 성공 ↓
[이미 위임 신청 여부 확인]
  ├─ 이미 신청 완료 → 기신청 안내 (수임인 정보 표시, 취소 가능)
  └─ 미신청 ↓
[위임 신청 화면]
  ├─ 특정인 위임: 수임인 이름 + 수임인 회원번호 입력 → 명단 실시간 검증
  └─ 의장 위임: 의장 선택
    ↓
[제출 확인 모달]
"위임을 신청하시겠습니까? 신청 후 취소는 위임 신청 기간 내에만 가능하며, 이후에는 취소가 불가합니다."
    ↓
[위임 신청 완료]
```

**위임 신청 기간과 투표 기간의 관계:**

| 단계 | 기간 |
|---|---|
| 위임 신청 기간 | `delegation_start_at` ~ `delegation_end_at` (투표 시작 전) |
| 투표 기간 | `start_at` ~ `end_at` |

- 두 기간은 겹치지 않도록 설계 (위임 신청 종료 후 투표 시작)
- 위임 신청 기간 중 투표 화면 접근 시 → "투표는 {start_at}에 시작됩니다." 안내

**위임 신청 규칙:**
- **1인 1건** 원칙 — 동일 투표에 위임은 1건만 허용
- **본인 위임 불가** — 본인 회원번호 입력 시 즉시 차단
- **수정 가능** — 기신청 건이 있는 상태에서 새 수임인 정보로 재제출하면 기존 건의 수임인 정보가 업데이트됨 (신규 건 생성 없이 기존 레코드 수정)
- **참석 신청과 상호 배타** — 참석 신청 중인 상태에서 위임 신청 시 확인 모달 표시 후 참석 신청 자동 취소 → 위임으로 전환
- **취소 시 소프트 삭제** — DB에서 삭제하지 않고 `status = '취소'`로 변경. 이력 보존하여 사후 문의 대응

**위임 취소 처리:**
- **기간 중 자가 취소:** 위임 신청 기간 종료 전까지 본인이 직접 취소 가능
- **기간 후 취소:** 순수 전자투표의 경우 기간 종료 후 취소 불가. 현장 총회 병행 투표의 경우에 한해 관리자가 A-09에서 직접 취소 처리 가능 (현장 출석 확인 후)

## 5-1. 전체 플로우 개요

```
[투표 링크 수신]
    ↓
[투표 유효성 확인]
  ├─ 기간 종료 → 종료 안내 페이지
  ├─ 아직 시작 안됨 → 대기 안내 페이지
  └─ 진행 중 → 인증 화면으로 이동
    ↓
[이니시스 간편인증]
  ├─ 인증 실패 → 실패 안내 + 재시도
  └─ 인증 성공 (이름, 전화번호, 생년월일 수집)
    ↓
[회원번호 입력]
    ↓
[1차 매칭: 이름+전화번호+회원번호]
  ├─ 성공 → 이미 투표 여부 확인
  └─ 실패 ↓
[2차 매칭: 이름+생년월일+회원번호]
  ├─ 성공 → 이미 투표 여부 확인
  └─ 실패 → 예외 요청 폼
    ↓
[이미 투표 여부 확인]
  ├─ 이미 완료 → 투표 완료 화면 (결과 미공개)
  └─ 미투표 → 투표 화면 입장
    ↓
[투표 진행]
    ↓
[제출 전 최종 확인]
    ↓
[투표 제출]
    ↓
[투표 완료 화면]
```

## 5-2. STEP별 상세 설명

### STEP 1. 투표 링크 접속

- **진입 경로:** 문자 / 알림톡 / 이메일로 발송된 URL
- **URL 구조:** `/vote/{voteId}` 또는 예외 링크 `/vote/{voteId}/exception-login?token=xxx`
- **투표 상태별 분기:**

| 투표 상태 | 화면 처리 |
|---|---|
| 준비중 | "아직 투표가 시작되지 않았습니다" + 시작 예정 시간 안내 |
| 진행중 | 정상 인증 화면으로 이동 |
| 종료 | "투표가 종료되었습니다" + 종료 시간 안내 |
| 존재하지 않음 | 404 오류 화면 |

### STEP 2. 이니시스 간편인증

- **진행 방식:** 이니시스 SDK 팝업 또는 임베드 형태
- **사용자 제공 정보:** 본인 명의 휴대폰으로 인증
- **수집 데이터:**

| 항목 | 설명 |
|---|---|
| authName | 인증된 이름 |
| phoneNumber | 인증된 휴대폰번호 (하이픈 제거, 01012345678 형태) |
| birthDate | 생년월일 (YYYYMMDD) |
| uniqueId | 이니시스 고유 인증 ID (중복 인증 방지용) |
| authTime | 인증 시간 |

- **오류 처리:**

| 오류 상황 | 사용자 안내 | 처리 |
|---|---|---|
| 인증 취소 | "인증을 취소하셨습니다. 다시 시도해주세요." | 재시도 버튼 표시 |
| 인증 실패 | "본인인증에 실패했습니다. 다시 시도해주세요." | 재시도 버튼 표시 |
| 통신 오류 | "통신 오류가 발생했습니다. 잠시 후 다시 시도해주세요." | 재시도 / 관리자 문의 |
| 타임아웃 (30초) | "인증 시간이 초과되었습니다." | 재시작 버튼 표시 |

### STEP 3. 회원번호 입력

- 이니시스 인증 성공 직후 노출
- 인증으로 수집된 이름을 마스킹 표시 (예: 홍*동)하여 본인 확인 유도
- 입력 필드: 회원번호 (텍스트, 필수)
- 확인 버튼 클릭 시 서버 매칭 요청

### STEP 4. 1차 회원 매칭

**매칭 기준:** 이름 + 전화번호 + 회원번호 (3개 모두 일치)

| 정규화 항목 | 규칙 |
|---|---|
| 전화번호 | 하이픈/공백 제거 (01012345678) |
| 이름 (한글) | 양쪽 공백 제거, 정확히 일치 |
| 이름 (영문) | 소문자 통일 후 비교 |
| 회원번호 | 공백 제거, 선행 0 유지 |

- **성공:** STEP 6 (중복 투표 확인)으로 이동
- **실패:** STEP 5 (2차 매칭)으로 이동

### STEP 5. 2차 회원 매칭

**매칭 기준:** 이름 + 생년월일 + 회원번호 (전화번호 변경 대비)

| 정규화 항목 | 규칙 |
|---|---|
| 생년월일 | YYYYMMDD 통일 (6자리 입력 시 19XX 처리) |
| 이름, 회원번호 | 1차와 동일 |

- **성공:** STEP 6으로 이동
- **실패:** STEP 5-1 (예외 요청)으로 이동

### STEP 5-1. 검증 실패 → 예외 요청 접수

**화면에 표시되는 안내:**
```
본인인증 정보가 회원 데이터와 일치하지 않습니다.
다음 중 하나의 경우일 수 있습니다:

1. 전화번호가 변경된 경우
2. 회원번호를 잘못 입력한 경우
3. 등록된 이름과 다른 경우 (개명 등)
4. 관리자 투표명단 등록 정보 오류

[ 다시 시도 ]   [ 관리자에 문의 ]
```

**예외 요청 폼 항목:**

| 항목 | 구분 | 설명 |
|---|---|---|
| 이름 | 자동 입력 (읽기전용) | 이니시스 수집값 |
| 현재 휴대폰번호 | 자동 입력 (읽기전용) | 이니시스 수집값 |
| 생년월일 | 자동 입력 (읽기전용) | 이니시스 수집값 |
| 회원번호 | 사용자 입력 | 재확인 입력 |
| 이전 휴대폰번호 | 선택 | 변경 전 번호 |
| 문제 원인 선택 | 선택 (체크박스) | 전화번호 변경 / 회원번호 오입력 / 개명 / 기타 — "기타" 선택 시 텍스트 입력 필드 표시 (직접 입력) |
| 추가 연락처 | 선택 | 관리자 답변 수신용 |

### STEP 6. 기투표 여부 확인

- **이미 투표 완료된 경우:** 투표 화면 입장 (투표 기간 내 수정 가능)
- **미투표:** 투표 화면 입장

### STEP 7. 투표 진행

- **투표 화면 진입 시 세션 생성** (투표 종료일시까지 유효, 2시간 비활성 시 간이 재확인)
- **1화면 1안건** 구조 — 한 번에 안건 1개만 표시, 이전/다음 버튼으로 순서 이동
- **상단 진행현황:** 안건 N개 중 M번째 + 완료된 안건 수 표시
- **위임받은 건이 있는 경우** 같은 화면에 본인 투표 카드 + 위임 카드(위임인별 1장)를 함께 표시

**화면 레이아웃:**
```
[안건 1 / 전체 N]  ←이전  다음→

━━━━━━━━━━━━━━━━━━━━━━━
안건 제목
안건 설명
━━━━━━━━━━━━━━━━━━━━━━━

┌─────────────────────┐
│ 내 투표             │  ← 본인 카드
│ ○ 선택지A           │
│ ○ 선택지B           │
│ ○ 기권              │
│         [제출]      │
└─────────────────────┘

┌─────────────────────┐
│ 홍*동님 위임 (1/3)  │  ← 위임 카드 (위임인명 마스킹)
│ ○ 선택지A           │
│ ○ 선택지B           │
│ ○ 기권              │
│         [제출]      │
└─────────────────────┘

┌─────────────────────┐
│ 김*수님 위임 (2/3)  │
│  ...                │
└─────────────────────┘
```

**안건별 상태 표시:**

| 상태 | 배지 색 | 배지 텍스트 | 비고 |
|---|---|---|---|
| 준비중 | 회색 | 대기중 | 선택 불가 |
| 열림 (미투표) | 파랑 | 진행중 | 선택 및 제출 가능 |
| 열림 (투표완료) | 초록 | 완료 | 재제출로 수정 가능 |
| 닫힘 (투표완료) | 회색 | 마감 | 수정 불가 |
| 닫힘 (미투표) | 옅은 빨강 | 미참여 | 미투표 자동 처리 |

### STEP 8. 투표 제출

- 카드별 "제출" 버튼 클릭 → 즉시 서버 제출 (본인 카드·위임 카드 각각 독립 제출)
- 제출 후 해당 안건이 열림 상태인 동안 "수정" 버튼으로 교체 → 재제출로 수정 가능
- 이미 제출된 카드는 선택값이 채워진 상태로 표시

### STEP 9. 투표 완료

- 완료 화면 표시: "투표에 참여해 주셔서 감사합니다."
- 투표 참여 일시 표시
- 투표 결과는 투표 종료 후 공지 예정 안내

### STEP 10. 세션 만료 처리

- 2시간 비활성으로 세션 만료 시 → 이름 + 회원번호 입력 팝업 표시 (이니시스 재호출 없음)
- 입력 일치 시 세션 재개 → 기존 투표 화면으로 복귀
- 불일치 시 "정보가 일치하지 않습니다." 인라인 메시지, 재입력 가능

# 6. 인증 및 권한 정책

## 6-1. 투표자 인증 정책

### 기본 인증 수단: 이니시스 간편인증

| 항목 | 내용 |
|---|---|
| 인증 방식 | 본인 명의 휴대폰 간편인증 (PASS, 카카오, 네이버 등) |
| 인증 기관 | 이니시스 |
| 수집 데이터 | 이름, 휴대폰번호, 생년월일 |
| 인증 유효 시간 | 30초 이내 완료 요구 |
| 재인증 | 인증 실패 시 재시도 가능 (횟수 제한 없음, 단 Rate Limit 적용) |

### 회원 매칭 정책

| 매칭 단계 | 기준 | 결과 |
|---|---|---|
| 1차 매칭 | 이름 + 전화번호 + 회원번호 | 성공 시 투표 입장 |
| 2차 매칭 | 이름 + 생년월일 + 회원번호 | 성공 시 투표 입장 |
| 매칭 실패 | — | 예외 요청 접수 |

### 쇼핑몰 회원 여부와 무관한 처리

- 이니시스 인증 통과 후 구글 스프레드시트 정회원 명단과만 매칭
- 쇼핑몰 회원 여부, 티켓 구매 여부는 신규 시스템에서 무관

### 중복 로그인 정책

- 동일 회원번호로 이미 세션이 존재하는 경우: 기존 세션 무효화 후 새 세션 생성
- 로그: "동일 계정 중복 로그인 발생" 기록

### 세션 정책

> ⚠️ **이니시스 인증 비용 이슈:** 이니시스 간편인증은 호출 1회당 비용이 발생한다. 회원이 투표 기간 내 위임 수정, 재확인 등을 위해 여러 번 접속할 경우 불필요한 인증 비용이 누적될 수 있다.  
> **단기 대응:** 인증 성공 후 세션을 투표 종료일시까지 유지하여 재인증 최소화.  
> **장기 권장:** 정회원 로그인 시스템 도입 → 이니시스는 최초 1회 신원 확인 용도로만 사용, 이후 ID/PW 로그인으로 전환 (Section 16 참조).

| 항목 | 내용 |
|---|---|
| 세션 유효기간 | 투표 종료일시까지 (재인증 비용 최소화 목적) |
| 세션 저장 | 서버 사이드 세션 (PHP Session 또는 DB 기반) |
| 쿠키 설정 | Secure + HttpOnly + SameSite=Strict |
| 비활성 만료 | 2시간 비활성 시 세션 만료 → 이름 + 회원번호 입력으로 세션 재개 (이니시스 재호출 없음) |

## 6-2. 관리자 인증 정책

| 항목 | 내용 |
|---|---|
| 로그인 방식 | ID + 비밀번호 |
| 비밀번호 정책 | 8자 이상, 영문+숫자+특수문자 조합 |
| 비밀번호 저장 | bcrypt 해싱 (cost factor 12 이상) |
| 세션 유효기간 | 4시간 (브라우저 닫기 시 만료) |
| 2차 인증 | 향후 도입 검토 (OTP 또는 이메일 인증) |

## 6-3. 예외 인증 정책 (1회용 링크)

> **적용 대상:** 이니시스 본인인증 자체가 불가한 경우에 한함 (타인 명의 기기, 해외 번호 등).  
> 단순 명단 데이터 오류(이름·전화번호·생년월일 오타 등)는 관리자가 스냅샷을 직접 수정하여 일반 인증 플로우로 처리.

| 항목 | 내용 |
|---|---|
| 발급 조건 | 관리자 승인 (스냅샷에서 대상 회원 직접 선택 후 발급) |
| 링크 유효기간 | 해당 투표 종료일시까지 (투표 기간 내 언제든 사용 가능) |
| 사용 횟수 | 1회 (이름+회원번호 확인 완료 및 세션 생성 시 즉시 만료) |
| 본인 확인 | 링크 진입 후 이름 + 회원번호 입력 → 스냅샷의 매핑된 회원 정보와 일치 시에만 통과 |
| 불일치 처리 | 이름 또는 회원번호 불일치 시 "입력 정보가 일치하지 않습니다." 표시, 링크 만료시키지 않음 (재시도 가능) |
| 토큰 길이 | 64자 (crypto.randomBytes(32).toString('hex')) |
| 로그 기록 | "관리자 예외 승인" 타입으로 별도 표시, 발급 관리자 + 대상 회원번호 함께 기록 |

# 7. 투표 유형 설계

## 7-0. 진행 방식 설계

투표 생성 시 **진행 방식**을 선택한다. 진행 방식에 따라 모수 산정 기준과 참석 신청 기능 활성화 여부가 달라진다.

| 진행 방식 | 설명 | 참석 신청 |
|---|---|---|
| 온라인 | 전체 기준인원이 온라인으로 투표 | 불필요 |
| 현장 | 현장 참석자만 투표 (총회 현장 진행) | 필요 |

### 모수 산정 기준

| 진행 방식 | 전체 모수 구성 |
|---|---|
| 온라인 | 실제 투표 참여자 수 (결과 분모) — 기준인원 명단 전체(대상자 수)는 참여율 산정 및 별도 표시 용도 |
| 현장 | 실제 출석 확정자 + 출석자에게 위임된 건 + 의장 귀속 위임 건 |

> **의장 귀속 위임:** 진행 방식에 무관하게 항상 모수에 포함된다.  
> **수임인 불참 처리 (현장 포함):** 특정인 위임에서 수임인이 현장에 미출석한 경우 해당 위임 건은 무효 처리 → 위임인·수임인 모두 모수에서 제외.

### 참석 신청 기간 (현장 포함 시)

현장이 포함된 진행 방식에서는 투표 생성 시 참석 신청 기간을 설정한다.

| 단계 | 기간 |
|---|---|
| 참석 신청 기간 | `attendance_start_at` ~ `attendance_end_at` |
| (선택) 위임 신청 기간 | `delegation_start_at` ~ `delegation_end_at` |
| 투표 기간 | `start_at` ~ `end_at` |

- 참석 신청은 회원이 U-18 화면에서 직접 신청
- 관리자는 A-10에서 신청 목록 조회 및 현장 출석 체크인 처리
- 출석 확정 후 모수가 확정됨 (투표 시작 후 관리자가 현장 체크인 완료 처리)

## 7-1. 투표 템플릿 구조

투표는 **템플릿 + 안건 유형** 조합으로 구성된다.

```
투표 (Vote)
  └─ 템플릿 선택 (임원선출 / 헌장개정 / 총회의결 / 커스텀)
       └─ 안건 (Agenda) 1..N
            └─ 안건 유형 선택 (찬반형 / 단일선택형 / 복수선택형 / 후보선택형 / 서술형 / 참석형 / 위임형)
                 └─ 선택지 (Choice) 1..N
```

## 7-2. 안건 유형별 설계

### ① 찬반형 (binary)

| 항목 | 내용 |
|---|---|
| 선택지 | 찬성 / 반대 / 기권 (고정) |
| 사용 예 | 헌장 개정, 총회 의결, 긴급 안건 |
| 검증 | 1개만 선택 가능 |
| 결과 표시 | 찬성 N명(X%) / 반대 N명(X%) / 기권 N명(X%) / 미투표 N명(X%) |

### ② 단일선택형 (single_choice)

| 항목 | 내용 |
|---|---|
| 선택지 | 관리자가 자유 구성 (2개 이상) |
| 사용 예 | 장소 선정, 일정 투표 |
| 검증 | 1개만 선택 가능 |
| 기권 옵션 | 선택적 추가 가능 |

### ③ 복수선택형 (multi_choice)

| 항목 | 내용 |
|---|---|
| 선택지 | 관리자가 자유 구성 |
| 사용 예 | 위원 선출 (복수 선발) |
| 검증 | 관리자가 최소/최대 선택 수 지정 |
| 오류 메시지 | "최소 N개, 최대 M개 선택해주세요." |

### ④ 후보선택형 (candidate)

| 항목 | 내용 |
|---|---|
| 선택지 | 후보자 카드 (직책, 사진, 이름, 소개, 공약 포함) |
| 사용 예 | 임원 선출, 회장/부회장 선거 |
| 지원 형태 | 단독 후보 (찬반) / 복수 후보 (1인 선택) / 팀 출마 (러닝메이트) |
| 개인 후보 항목 | **직책** / 이름 / 사진 / 소개 / 공약 |
| 팀 후보 항목 | 팀명 / **팀원별 직책+이름+개인사진** / **팀 포스터(선택)** / 소개 / 공약 |
| 기권 옵션 | 항상 포함 (선거 유효성 기준) |
| 후보 상세 | 모달로 전체 소개 확인 가능 |

> ⚠️ **단일 후보 등록 시 필수 처리 정책 *(v1.2 신규)*:**  
> 후보가 1인(또는 1팀)만 등록된 경우 투표 화면은 **찬반 방식**으로 표시된다. 이 경우에도 후보자 또는 팀의 **사진, 직책, 소개, 공약** 등 후보 정보를 반드시 등록해야 하며, 미등록 시 안건 저장이 불가하다. 관리자가 후보 등록 없이 단독 찬반으로만 처리하는 것은 허용하지 않는다.

**후보 선택형 화면 차이:**

| 형태 | 등록 방식 | 투표 선택지 구성 |
|---|---|---|
| 단독 후보 (개인) | 1명 등록 — 사진·직책·소개·공약 필수 | 찬성 / 반대 / 기권 |
| 단독 후보 (팀) | 1팀 등록 — 팀원별 사진·직책 필수, 팀 소개·공약 필수 | 찬성 / 반대 / 기권 |
| 복수 후보 | 2명 이상 등록 | [직책] 후보A / [직책] 후보B / ... / 기권 |
| 팀 출마 | 2팀 이상 등록 | 팀A (직책 후보1 + 직책 후보2) / 팀B / ... / 기권 |

**후보자 카드 표시 구성 (개인):**
```
[사진]
직책: 회장
이름: 홍길동
```

**후보자 카드 표시 구성 (팀):**
```
[팀 포스터] (등록 시 표시, 미등록 시 생략)
팀명: ○○팀
  [개인사진] 회장 홍길동
  [개인사진] 사무국장 임꺽정
```

### ⑤ 서술형 (essay)

| 항목 | 내용 |
|---|---|
| 입력 | 자유 텍스트 (textarea) |
| 사용 예 | 의견 수렴, 건의사항 |
| 글자 수 | 관리자가 최대 글자 수 설정 가능 |
| 결과 | 텍스트 목록으로 표시 (통계 없음) |
| 익명 여부 | 관리자가 설정 |

### ⑥ 참석형 (attendance)

| 항목 | 내용 |
|---|---|
| 선택지 | 참석 / 불참 |
| 사용 예 | 총회 참석 신청, 행사 참여 여부 |
| 결과 | 참석 N명 / 불참 N명 |

## 7-3. 안건 상태 전환 규칙

```
준비중 ──→ 열림 ──⇄── 닫힘
   │          │
   ↓          ↓
  제외       제외 (닫힘 → 제외 불가)
```

| 전환 | 처리 내용 |
|---|---|
| 준비중 → 열림 | 투표 시작 |
| 열림 → 닫힘 | 미투표자 자동 미투표 처리 |
| 닫힘 → 열림 | 미투표 처리 초기화, 재투표 가능 |
| 준비중/열림 → 제외 | 2단계 확인 팝업, 기투표 데이터 내부 보존 |
| 제외 → 재포함 | 불가 (신규 등록 필요) |

## 7-4. 의결 기준 *(v1.2 신규)*

투표 결과 산정은 아래 원칙을 기본으로 한다. 투표 유형별로 적용 기준이 다르며, 결과 조회 화면(A-12)에서 해당 기준에 따른 가결/부결/당선 여부를 자동으로 표시한다.

### 기본 원칙: 다수결

| 항목 | 내용 |
|---|---|
| 적용 대상 | 임원 선출 (복수 후보), 단일선택형, 복수선택형 |
| 기준 | **실제 투표 참여자 수** 기준 가장 많은 표를 얻은 선택지가 선출/결정됨 |
| 동점 처리 | ⚠️ 고객사 확인 필요 (재투표 / 상위 기관 결정 / 기타) |

### 찬반 의결 기준: 유효표 과반수 이상 찬성

| 항목 | 내용 |
|---|---|
| 적용 대상 | 찬반형 안건 (헌장 개정, 긴급 안건, **단독 후보 찬반** 포함) |
| 유효표 정의 | 실제 참여자 중 기권표를 제외한 찬성 + 반대 표수 |
| 가결 기준 | **찬성 수 > 유효표의 과반수 (50% 초과)** |
| 예시 | 대상자 100명, 실제 참여 70명 중 찬성 40명, 반대 20명, 기권 10명 → 유효표 60표 중 찬성 40표(66.7%) → **가결** |
| 부결 기준 | 찬성 수 ≤ 유효표의 50% 이하 |
| 비참여자 처리 | 참여율(대상자 대비) 집계에만 반영 — 결과 분모에서는 제외 |

> ⚠️ **정족수(쿼럼) 기준은 별도 확인 필요:** 유효 투표 성립을 위한 최소 참여율(예: 전체 유권자의 1/3 이상 참여)을 적용할지 여부는 고객사 확인 후 기획서에 반영한다.

### 결과 표시 예시

**찬반형 안건 결과:**
```
대상자: 100명 | 실제 참여자: 70명 (참여율 70%)

찬성: 40명 (57.1%)  ←── 유효표 기준
반대: 20명 (28.6%)
기권: 10명 (14.3%)  ←── 유효표 계산 제외

유효표: 60표 | 가결 기준: 31표 초과
→ [가결] ✅
```

**단독 후보 찬반 결과:**
```
[회장] 홍길동
대상자: 100명 | 실제 참여자: 80명 (참여율 80%)
찬성: 55명 (68.8%) | 반대: 25명 (31.3%) | 기권: 0명
유효표: 80표 | 찬성 55표(68.8%) → [당선] ✅
```

## 7-5. 임원선출 표준 일정 샘플 *(v1.2 신규)*

임원선출은 아래 7단계로 진행된다. **전자투표 시스템이 관여하는 구간은 6~7번** (투표 기간 및 개표·결과 공고)이며, 1~5번은 홈페이지 공고 등 외부 채널을 통해 진행된다.

| 단계 | 일정 예시 | 처리 주체 | 시스템 관여 |
|---|---|---|---|
| 1 | 2025.01.19 | 선관위 기준일 확정 | 선관위 | ❌ |
| 2 | 2025.01.19 ~ 2025.01.30 오후 6시 | 후보 등록 기간 | 선관위 수동 처리 (시스템화 X) | ❌ |
| 3 | 2025.01.31 | 후보자 공고 및 선거운동 시작 (오전 9시부터 가능) | 선관위/홈페이지 | ❌ |
| 4 | 2025.02.03 | 등록자료 보완 마감 기한 | 선관위 | ❌ |
| 5 | 2025.02.04 오후 6시 | 선거운동 마감 | 선관위 | ❌ |
| **6** | **2025.02.05 오전 9시 ~ 2025.02.07 오후 6시** | **투표 기간 (전자투표)** | **전자투표 시스템** | ✅ |
| **7** | **2025.02.07 오후 6시 이후** | **개표 및 선거결과 공고 (당선인 즉시 임기 시작)** | **전자투표 시스템 + 선관위** | ✅ |

**시스템 운영 포인트:**
- 투표 기간(6번) 시작 전 관리자(선관위)가 후보자 정보를 시스템에 사전 등록 완료해야 함
- 투표 기간 중 추가 명부 수정이 발생할 수 있으므로, 관리자는 투표 시작 직전까지 기준인원 명단 최종 확인 필요
- 개표(7번)는 투표 종료 즉시 결과 조회 화면(A-12)에서 자동 집계 확인 가능 — 결과 공표 방식(자동/수동)은 투표 생성 시 선택
- 당선인은 결과 공표 시점부터 즉시 임기 시작 → 당선증 발급 기능(A-20) 활용 가능

### U-07. 투표 메인 (1화면 1안건)

| 항목 | 내용 |
|---|---|
| 목적 | 안건별 투표 진행 — 한 번에 1개 안건 표시, 이전/다음으로 이동 |
| 주요 컴포넌트 | 상단 진행현황 바, 안건 정보, 투표 카드(본인+위임), 이전/다음 버튼 |
| 상태값 | 안건별 상태 (준비중/열림/닫힘) + 카드별 투표 완료 여부 |
| 사용자 액션 | 선택지 클릭, 카드별 제출, 이전/다음 이동, 후보 상세 보기 |

**화면 구성:**
```
┌─────────────────────────────────────┐
│ 안건 2 / 5   [████░░░░░] 2/5 완료  │  ← 상단 진행현황
└─────────────────────────────────────┘

  [파랑 배지: 진행중]
  제1호 안건: 헌장 개정안
  2025년 정관 제3조 개정에 관한 건...

  ┌─────────────────────────────────┐
  │ 내 투표                [완료 ✓] │  ← 이미 제출된 경우 선택값 표시
  │ ● 찬성                          │
  │ ○ 반대                          │
  │ ○ 기권                          │
  │                      [수정]     │
  └─────────────────────────────────┘

  ┌─────────────────────────────────┐
  │ 홍*동님 위임 (1/3)              │
  │ ○ 찬성                          │
  │ ○ 반대                          │
  │ ○ 기권                          │
  │                      [제출]     │
  └─────────────────────────────────┘

  ┌─────────────────────────────────┐
  │ 김*수님 위임 (2/3)              │
  │  ...                            │
  └─────────────────────────────────┘

  [← 이전 안건]              [다음 안건 →]
```

**안건 상태 배지:**

| 안건 상태 | 배지 색 | 배지 텍스트 | 설명 |
|---|---|---|---|
| 열림 + 투표 완료 | 초록 | 완료 | 투표 기간 내 수정 가능 |
| 열림 + 미투표 | 파랑 | 진행중 | 지금 투표 가능 |
| 닫힘 + 투표 완료 | 회색 | 마감 | 종료됨, 투표 참여함 |
| 닫힘 + 미투표 | 옅은 빨강 | 미참여 | 종료됨, 미투표 자동 처리 |

**투표 카드 규칙:**
- 본인 카드 항상 최상단
- 위임 카드는 위임인명 마스킹 표시 (예: 홍*동), 위임 순서대로 표시
- 카드별 제출 독립 — 본인 제출·위임 제출은 각각 별개로 처리
- 제출된 카드: 선택값 유지 + "수정" 버튼 표시 (열림 상태 안건에 한함)
- 닫힘 상태 안건: 모든 카드 읽기전용, 제출/수정 버튼 없음
- 의장 귀속 위임: 이 화면에 카드 미표시 — A-14에서 최고관리자가 별도 처리

**진행현황 계산:**
```
전체(N) = 준비중 + 열림 + 닫힘 안건 수 (제외 안건 미포함)
완료(M) = 본인이 제출 완료한 안건 수 + 닫힘 상태 안건 전체
```

**안건 이동 방식 — 이전/다음 + 드롭다운:**
```
[← 이전]  [▼ 안건 2/5 헌장 개정안]  [다음 →]
              ↓ 클릭 시 드롭다운 펼침
           ┌──────────────────────────┐
           │ ✓ 1. 개회 선언    [마감] │
           │ → 2. 헌장 개정안  [진행중]│  ← 현재 안건
           │   3. 임원 선출    [진행중]│
           │   4. 예산안 심의  [대기중]│
           │   5. 폐회         [대기중]│
           └──────────────────────────┘
```
- 드롭다운 항목 클릭 → 해당 안건으로 바로 이동
- 각 항목에 안건 번호 + 제목 + 상태 배지 표시
- 첫 번째 안건에서 "이전" 버튼 비활성, 마지막 안건에서 "다음" 버튼 비활성

**모바일 고려사항:**
- 카드 세로 스크롤, 이전/다음 버튼 하단 고정 (sticky)
- 선택지 버튼 최소 44px 터치 영역
- 오프라인 대비 재시도 안내

### U-08. 후보자 소개 모달 (후보선택형 전용)

| 항목 | 내용 |

| 트리거 | 후보 카드의 "소개 보기" 버튼 클릭 |
| 표시 내용 | 직책, 사진, 이름, 소개글, 공약 / 팀 출마 시: 팀 포스터(선택) + 팀명 + 팀원별 직책·이름·개인사진 |
| 닫기 | X 버튼 또는 모달 외부 클릭 |
| 모바일 | 바텀 시트(Bottom Sheet) 형태 권장 |

### U-09. 투표 완료

| 항목 | 내용 |

| 버튼 | 없음 |

| 투표확인증 | 이번 구축 범위 제외 — 추후 확장 시 발급 버튼 노출 예정 |

### U-15. 투표확인증 발급 *(Kevoting 벤치마킹 — 이번 구축 범위 제외, 추후 확장)*

> ⚠️ **이번 구축 범위에서 제외된 화면입니다.** 추후 확장 구축 시 아래 설계를 기준으로 구현합니다.

| 항목 | 내용 |
|---|---|
| 목적 | 투표 참여 사실을 증명하는 개인 고유 확인증 발급 |
| 진입 조건 | 투표 완료 + 관리자가 발급 기능 활성화 + 투표 종료 후 OR 즉시 (설정에 따름) |
| 경로 | `/vote/{voteId}/certificate?token={voteToken}` |
| 주요 내용 | 투표명, 투표자 이름(마스킹), 참여 일시, 고유 확인 코드, QR코드 |
| 고유 코드 | 32자 hex 토큰 (위조 검증 불가 — 참여 증명용이므로 공개 검증은 미제공) |
| 다운로드 | PDF 저장 또는 이미지 저장 버튼 (브라우저 print API 활용) |
| 재발급 | 동일 투표 완료자라면 세션 유지 중 재접근 가능 |
| 모바일 | 전체 화면 세로 카드 형태, 인쇄 친화적 레이아웃 |
| 고객 확인 | 발급 기능 활성화 여부, 발급 시점(투표 완료 즉시 or 투표 종료 후) 결정 필요 |

### U-16. 위임 신청 / 확인 *(추후 확장)*

> ⚠️ **이번 구축 범위에서 제외된 화면입니다.** 정기총회(위임 포함)는 2단계 확장 구축 시 아래 설계를 기준으로 구현합니다.

| 항목 | 내용 |
|---|---|
| 목적 | 투표 단위 위임 신청·수정·취소 / 나에게 위임된 건 확인 |
| 경로 | `/vote/{voteId}/delegate` |
| 진입 조건 | 이니시스 인증 + 회원 매칭 성공 (기간 외에도 접근 가능, 단 신청·수정·취소는 기간 중만 활성) |

**화면 구성 — 인증 후 두 영역으로 표시:**

---

**① 내 위임 신청 현황 (상단)**

| 상태 | 표시 내용 |
|---|---|
| 미신청 + 기간 중 | 위임 신청 폼 표시 |
| 미신청 + 기간 외 | "위임 신청 기간이 아닙니다." 안내만 표시 |
| 신청 완료 + 기간 중 | 기신청 정보 표시 + "수정" / "취소" 버튼 |
| 신청 완료 + 기간 종료 | 기신청 정보 표시 (읽기전용, 버튼 없음) |
| 취소 완료 | "취소된 위임 신청입니다." + 취소 일시 표시 |

**기신청 정보 표시 항목:**
- 수임인명 (특정인) 또는 "의장"
- 신청 일시
- 현재 상태 (유효 / 취소)

**수정 처리:**  
"수정" 버튼 클릭 → 신청 폼이 현재 값으로 채워진 상태로 펼쳐짐 → 재제출 시 기존 건의 수임인 정보 업데이트 (기존 레코드 수정, 신규 건 생성 없음)

**위임 신청 폼 (신규 또는 수정):**

| 항목 | 구분 | 설명 |
|---|---|---|
| 위임 유형 선택 | 라디오 버튼 | 특정인 위임 / 의장 위임 |
| 수임인 이름 | 입력 (특정인 선택 시) | 필수 |
| 수임인 회원번호 | 입력 (특정인 선택 시) | 필수 |
| [확인] 버튼 | 수임인 정보 입력 후 클릭 | 검증 실행 |
| 검증 결과 | 인라인 표시 | "확인되었습니다." / "일치하는 회원이 없습니다." |

> **수임인 검증 정책 — 보안 중요**  
> 이름 + 회원번호를 **두 값 모두** 입력 후 [확인]을 눌러야 검증이 진행된다.  
> 서버는 기준인원 명단에서 두 값 일치 여부(bool)만 반환한다.  
> 자동완성·회원 검색·부분 조회는 일절 제공하지 않는다.  
> → 수임인의 정보를 이미 알고 있는 경우에만 위임이 가능하도록 하여 회원번호 남용·유출을 차단한다.

**유효성 규칙:**
- 본인 회원번호 입력 시 검증 전에 즉시 차단: "본인에게는 위임할 수 없습니다."
- 이미 유효한 위임 신청이 있는 상태에서 재제출 → 기존 건 수정 (수임인 정보 업데이트, 신규 건 생성 없음)
- **참석 신청이 이미 완료된 상태에서 위임 신청 시** → 제출 버튼 클릭 전 충돌 안내 모달 표시:
  ```
  "현재 참석 신청이 되어 있습니다.
  위임을 신청하면 참석 신청이 자동으로 취소됩니다.
  계속하시겠습니까?"
  ```
  확인 시: 참석 신청 소프트 취소 → 위임 신청 진행

**제출 버튼 활성 조건:**
- 의장 위임: 라디오 선택만으로 활성
- 특정인 위임: 수임인 검증 통과 후 활성

**제출 확인 모달:**
```
[특정인 위임 — 신규 신청]
"{수임인명}님께 위임하시겠습니까?
신청 후 취소는 위임 신청 기간({delegation_end_at}) 내에만 가능하며, 이후에는 취소가 불가합니다."

[특정인 위임 — 수정]
"{수임인명}님으로 수임인을 변경하시겠습니까?
취소는 위임 신청 기간({delegation_end_at}) 내에만 가능합니다."

[의장 위임 — 신규 신청]
"의장에게 위임하시겠습니까?
신청 후 취소는 위임 신청 기간({delegation_end_at}) 내에만 가능하며, 이후에는 취소가 불가합니다."

[의장 위임 — 수정]
"의장으로 수임인을 변경하시겠습니까?
취소는 위임 신청 기간({delegation_end_at}) 내에만 가능합니다."
```

**취소 확인 모달:**
```
"위임 신청을 취소하시겠습니까?
취소 후에도 이 화면에서 취소 이력을 확인할 수 있습니다."
```

**② 나에게 위임된 건 (하단)**

인증한 회원이 수임인으로 등록된 위임 건 전체를 표시한다.  
기간 중이든 기간 후든 항상 조회 가능.

| 컬럼 | 내용 |
|---|---|
| 위임인명 | — |
| 신청 일시 | — |
| 상태 | **유효** / **취소** |

- 취소된 건은 회색 처리 + "취소" 배지 표시 (삭제하지 않음, 이력 보존)
- 위임받은 건이 없으면: "현재 나에게 위임된 건이 없습니다."
- 위임인명은 표시하되, 회원번호 등 상세 정보는 노출하지 않음

### U-17. 위임 신청 완료 *(추후 확장)*

> ⚠️ **이번 구축 범위에서 제외된 화면입니다.** 정기총회(위임 포함)는 2단계 확장 구축 시 아래 설계를 기준으로 구현합니다.

| 항목 | 내용 |
|---|---|
| 표시 내용 | "위임 신청이 완료되었습니다." |
| 추가 정보 | 수임인 이름 (특정인 위임) 또는 "의장" / 신청 일시 |
| 안내 문구 | "투표 기간({start_at} ~ {end_at}) 중에는 직접 투표가 불가합니다." |
| 버튼 | 없음 |

### U-18. 참석 신청 *(현장 포함 투표에서만 노출)*

| 항목 | 내용 |
|---|---|
| 노출 조건 | 진행 방식이 현장인 투표에 한함 |
| 접근 경로 | 투표 진입 화면(U-01) 또는 별도 링크 |
| 인증 | 이니시스 간편인증 + 회원번호 매칭 (일반 투표 진입과 동일) |

**화면 구성:**

| 상태 | 표시 내용 |
|---|---|
| 신청 기간 중 + 미신청 | 참석 신청 버튼 표시 |
| 신청 기간 중 + 신청 완료 | 신청 확인 정보 + "신청 취소" 버튼 |
| 신청 기간 종료 + 신청 완료 | 신청 확인 정보 (읽기전용) |
| 신청 기간 종료 + 미신청 | "참석 신청 기간이 종료되었습니다." 안내 |

**신청 확인 정보 표시 항목:**
- 투표명
- 신청 일시
- 현재 상태 (신청 완료 / 취소)

**참석 신청 확인 모달:**
```
[위임 신청 없는 경우]
"현장에 참석 신청하시겠습니까?
참석 신청 기간({attendance_end_at}) 내에 취소할 수 있습니다."

[위임 신청이 이미 완료된 경우]
"현재 위임 신청이 되어 있습니다.
참석 신청을 하면 위임 신청이 자동으로 취소됩니다.
계속하시겠습니까?"
확인 시: 위임 신청 소프트 취소 → 참석 신청 진행
```

**신청 취소 확인 모달:**
```
"참석 신청을 취소하시겠습니까?
취소 후 현장 참석이 불가합니다."
```

### U-19. 인증 불가 문의 폼

이니시스 본인인증 자체가 불가한 회원이 관리자에게 문의를 접수하는 화면.  
이니시스 인증 없이 직접 정보를 입력하며, 접수 후 관리자가 이메일 및 A-13에서 확인한다.

**폼 항목:**

| 항목 | 구분 | 설명 |
|---|---|---|
| 이름 | 필수 입력 | 본인 이름 직접 입력 |
| 회원번호 | 필수 입력 | 본인 회원번호 직접 입력 |
| 연락처 (이메일 또는 전화번호) | 필수 입력 | 관리자 답변 수신용 |
| 문의 내용 | 필수 입력 | 인증이 불가한 이유 및 상황 설명 (textarea) |

**제출 시 처리:**
- 접수 내용을 DB 저장 (`inquiry_type = auth_unable`)
- 관리자 이메일로 새 문의 접수 알림 자동 발송
- U-20 접수 완료 화면으로 이동

## 9-2. 주요 화면별 기능 상세

### A-04. 투표 생성/수정

**기본 정보:**
- 투표명, 투표 유형(템플릿), 기간(시작·종료 일시), 설명

**기준인원 명단 설정:**

| 설정 항목 | 설명 |
|---|---|
| 스프레드시트 ID | 시스템 전역 설정에서 관리 (투표별로 변경 불가) |
| 기준 시트명 | 해당 투표에서 사용할 시트명 입력 (예: 정회원, 임원, 이사회) |

- 시트명 입력 후 [연결 확인] 버튼 클릭 → 해당 시트 존재 여부 및 필수 컬럼(이름·회원번호·휴대폰번호) 유효성 즉시 검증
- 동일 기간 복수 투표가 열려도 시트명이 다르면 각각 독립된 기준인원 적용
- 저장 시 지정한 시트 기준으로 스냅샷 자동 생성 (초기 고정)
- 이후 관리자가 [구글 시트 재동기화] 버튼으로 최신 명단을 다시 불러와 스냅샷 갱신 가능 (2단계 확인 + 사유 입력 + 감사 로그)
- 투표 `열림` 상태 중 재동기화 시 추가·수정만 반영 — 시트에서 제거된 회원은 스냅샷 삭제 없이 투표권 유지

**진행 방식 설정:**

| 설정 항목 | 옵션 | 기본값 |
|---|---|---|
| 진행 방식 | 온라인 / 현장 | 온라인 |

- 현장 선택 시 참석 신청 기간 설정 영역 표시

**참석 신청 기간 설정 (현장 포함 시만 표시):**

| 설정 항목 | 설명 |
|---|---|
| 참석 신청 시작 일시 | 필수 입력 |
| 참석 신청 종료 일시 | 투표 시작 일시 이전이어야 함 |

- 참석 신청 종료 일시가 투표 시작 일시보다 늦으면 저장 불가 + 인라인 오류 표시

**위임 신청 기간 설정:**

| 설정 항목 | 설명 |
|---|---|
| 위임 신청 기능 | 사용 / 미사용 (기본: 미사용) |
| 위임 신청 시작 일시 | 사용 선택 시 필수 입력 |
| 위임 신청 종료 일시 | 투표 시작 일시 이전이어야 함 (유효성 검증) |

- 위임 신청 종료 일시가 투표 시작 일시보다 늦으면 저장 불가 + 인라인 오류 표시
- 위임 신청 기간이 설정된 투표는 투표 상세 화면에 "위임 신청 링크 복사" 버튼 제공

**운영 설정 (Kevoting 벤치마킹 반영):**

| 설정 항목 | 옵션 | 기본값 |
|---|---|---|
| 시작 방식 | 자동(예약 시작) / 수동(관리자 직접 시작) | 자동 |
| 결과 공표 방식 | 종료 즉시 자동 공표 / 관리자 수동 공표 | 수동 공표 |
| 투표확인증 발급 | 활성 / 비활성 | **이번 구축 범위 제외 (추후 확장)** |
| 확인증 발급 시점 | 투표 완료 즉시 / 투표 종료 후 | **이번 구축 범위 제외 (추후 확장)** |
| 기명/무기명 | 기명(투표자 식별 가능) / 무기명(집계만, 개인 식별 불가) | 기명 |
| 기권 처리 방식 | 안건별 개별 기권 버튼 / 투표 시작 시 일괄 기권 선언 | 개별 기권 |
| 당선증 발급 | 활성 / 비활성 (후보선택형 안건 존재 시만 표시) | 비활성 |

> ⚠️ **기명/무기명** 활성화 여부는 고객사 확인 필요 — 섹션 17 참조

**일시정지 기능:**
- 투표 진행 중 관리자가 일시정지 가능
- 일시정지 시: 투표자 화면에 "현재 투표가 일시 중단되었습니다. 잠시 후 다시 시도해주세요." 표시
- 일시정지 해제 시: 자동 복귀 (투표자 재인증 불필요, 세션 유지)
- 일시정지 이력 감사 로그 기록

### A-05. 투표 상세 (안건 관리)

**상단 정보:**
- 투표명, 유형, 기간, 상태
- 회원 명단 동기화 상태 및 버튼

**안건 목록 테이블:**

| 컬럼 | 내용 |
|---|---|
| 순서 | ▲/▼ 버튼 (열림 상태 안건은 비활성) |
| 안건명 | 클릭 시 상세 |
| 유형 | 찬반/단일/복수/후보 등 |
| 상태 | 준비중/열림/닫힘/제외 |
| 투표 현황 | X명 / Y명 참여 |
| 상태 변경 | 토글 버튼 |
| 액션 | 수정, 제외 |

**제외 처리:**
1. "제외" 클릭
2. 1차 확인: "제외 시 복구가 불가합니다. 계속하시겠습니까?"
3. 2차 확인: "정말 제외하시겠습니까?" (재확인)
4. 제외 처리 → 안건 목록 최하단 이동, 회색 배경 + "제외" 배지

**제외된 안건 표시:**
- 관리자 화면: 목록 하단, 회색 배경, 제외 배지, 기투표 건수만 표시 (비율 미표시)
- 투표자 화면: 표시하지 않음
- 결과 CSV: 포함하지 않음

### A-08. 회원 명단 관리

**기능:**
- 구글 스프레드시트 동기화 (API 또는 CSV 업로드)
- 동기화 시점 표시 (마지막 동기화: YYYY-MM-DD HH:MM)
- 회원 목록 조회 (이름 / 회원번호 / 전화번호 / 이메일 / 생년월일)
- 검색 및 정렬 (이름, 회원번호)
- CSV 다운로드 (전체 명단)
- 개별 회원 정보 수정 (투표 생성 후에도 허용 — 2단계 확인 + 수정 사유 필수 입력 + 감사 로그 기록)
- 개별 회원 추가 (투표 진행 중에도 선관위 재량으로 추가 가능 — 2단계 확인 + 추가 사유 필수 입력 + 감사 로그 기록)
- 개별 회원 삭제 (투표 진행 중에도 허용 — 단, 아래 삭제 정책 준수 필수)

**기준인원 명단 마감일 유동 적용 *(v1.2 신규)*:**  
기준인원 명단의 마감일은 선관위가 유동적으로 결정하며, 투표 진행 중에도 관리자가 명단을 추가하거나 수정할 수 있다. 단, 모든 변경은 2단계 확인 + 사유 입력 + 감사 로그 기록을 반드시 거쳐야 한다.

**명단 삭제 정책 *(v1.2 신규)*:**

> ⚠️ **삭제는 가급적 하지 않도록 강력히 권고합니다.**  
> 정보 오류 정정이 목적인 경우, 삭제 후 재등록보다 **수정(이름·전화번호·생년월일 등 교정)을 우선 사용**해주세요.

| 상황 | 처리 결과 |
|---|---|
| 아직 투표하지 않은 회원 삭제 | 이후 모수 및 투표 참여에서 제외 |
| **이미 투표를 완료한 회원 삭제** | **투표 결과(vote_logs)에는 영향 없음 — 기 투표 데이터는 영구 보존** |
| 삭제된 회원의 기 투표 데이터 | 민원 발생 시 감사 로그 및 vote_logs 통해 확인 가능 |

삭제 처리 시 UI에 아래 안내 문구를 필수 표시한다:
```
⚠️ 회원 삭제 전 반드시 확인하세요.

이미 투표를 완료한 회원을 삭제해도 해당 투표 결과는 유지됩니다.
삭제 후에는 해당 회원이 모수에서 제외됩니다.

삭제는 가급적 피하고, 정보 오류 수정은 [수정] 기능을 사용해주세요.

삭제 사유: [필수 입력 ______________]
```

**동기화 방식 선택:**

| 방식 | 설명 |
|---|---|
| Google Sheets API | 스프레드시트 ID(전역 설정) + 투표별 시트명 지정으로 자동 연동 (권장) |
| CSV 수동 업로드 | 관리자가 CSV 파일 직접 업로드 (시트 분리가 불필요한 경우) |

**시트명 기반 연동 구조:**
```
구글 스프레드시트 (전역 1개)
├── 정회원          ← 전체 정회원 대상 투표에서 지정
├── 임원            ← 임원 대상 투표에서 지정
├── 이사회          ← 이사회 투표에서 지정
└── (추가 시트...)
```
- 동일 시점에 여러 투표가 열려도 시트명이 다르면 스냅샷이 각각 독립적으로 생성됨

**명단 관리 방식 (혼합):**

| 방법 | 시점 | 설명 |
|---|---|---|
| 초기 스냅샷 | 투표 생성 시 | 지정한 구글 시트 기준으로 명단 자동 스냅샷 → DB 저장 |
| 구글 시트 재동기화 | 언제든지 | [재동기화] 버튼 클릭 → 구글 시트 최신 명단으로 스냅샷 갱신. **`열림` 상태에서는 추가·수정만 반영, 삭제 차단** (투표권 보호) |
| 개별 수정/추가 | 언제든지 | 스냅샷에서 특정 회원 정보를 직접 수정·추가 |
| 개별 삭제 | `준비중` 상태에서만 | 가급적 지양 — 기투표 데이터는 vote_logs에 영구 보존 |

> ⚠️ **명단 변경 정책:** 재동기화·수정·추가 모두 "정말 진행하시겠습니까?" 2단계 확인 팝업 → 사유 필수 입력 → 감사 로그 자동 기록 순으로 처리. **투표 `열림` 상태에서 재동기화 시 삭제는 반영되지 않으며**, 삭제가 필요한 경우 `닫힘` 이후 처리를 권장한다. 모든 변경 이력은 A-15 감사 로그에서 조회 가능.

**명부 업로드 실패 목록 *(Kevoting 벤치마킹)*:**

CSV 업로드 또는 Google Sheets 동기화 시 형식 오류가 있는 행은 업로드를 중단하지 않고 **정상 행만 처리 후 실패 목록을 별도 표시**한다.

| 컬럼 | 내용 |
|---|---|
| 행 번호 | 원본 파일 기준 행 번호 |
| 이름 | 실패 행의 이름 필드 원문 |
| 회원번호 | 실패 행의 회원번호 원문 |
| 실패 사유 | 필수값 누락 / 형식 오류 / 중복 행 / 회원번호 형식 불일치 등 |

**처리 흐름:**
1. 업로드 완료 후 실패 행이 있으면 경고 배너 표시: "N개 행이 등록되지 않았습니다."
2. "실패 목록 보기" 버튼 → 실패 행 테이블 펼침
3. 관리자가 원본 파일을 수정 후 재업로드 또는 개별 수동 추가
4. 실패 행은 스냅샷에 포함되지 않으므로, 모수 계산 전 반드시 확인 필요

### A-09. 위임 관리 *(추후 확장)*

> ⚠️ **이번 구축 범위에서 제외된 화면입니다.** 정기총회(위임 포함)는 2단계 확장 구축 시 아래 설계를 기준으로 구현합니다.

- 총 위임 신청 수: N건 (유효 N건 / 무효 N건)

| 컬럼 | 내용 |

| 컬럼 | 내용 |

**컬럼 정렬:** 모든 컬럼 헤더 클릭 시 오름차순/내림차순 전환, ▲/▼ 표시

### A-10. 참석자 관리 *(현장 포함 투표에서만 활성)*

**기능:**
- 참석 신청 목록 조회 (신청자 이름, 회원번호, 신청 일시, 상태)

- 위임 건 연동 표시 — 출석자에게 위임된 건 수 표시

**참석 신청 목록 테이블:**

| 컬럼 | 내용 |
|---|---|
| 이름 | — |
| 회원번호 | — |
| 신청 일시 | — |
| 상태 | 신청 완료 / 취소 / 출석 확인 |
| 위임 건수 | 해당 출석자에게 위임된 유효 건 수 |

**모수 현황 요약 (상단 카드):**
```
현장 출석 확정: N명
└─ 의장 귀속 위임: N건
└─ 출석자 특정인 위임: N건
```

### A-14. 의장 위임 투표 행사 *(추후 확장 — 최고관리자 전용)*

> ⚠️ **이번 구축 범위에서 제외된 화면입니다.** 정기총회(위임 포함)는 2단계 확장 구축 시 아래 설계를 기준으로 구현합니다.

의장에게 귀속된 위임 건 전체를 최고관리자(super_admin)가 안건별로 선택지 % 배분 방식으로 행사한다.  
선거관리자 이하 권한으로는 접근 불가.

**화면 구성:**

상단에 "의장 귀속 총 위임: N건" 표시 후, 안건별로 선택지 % 입력 UI 제공.

```
[안건명]  의장 귀속 위임: N건

찬성  [ 33.33 ]%  →  3명  (반올림 적용)
반대  [ 33.33 ]%  →  3명
기권  [ 33.34 ]%  →  4명  ← 보정 적용됨
합계: 100.00%  ✅

* 반올림 후 합계 불일치 시 최대 % 선택지에 자동 가감됩니다.
* 개별 표의 선택지 배정은 시스템이 무작위로 처리합니다.

[투표 행사] 버튼
```

**입력 규칙:**
- % 입력: 소수점 2자리까지 허용 (예: 33.33%)
- 각 선택지 % 합계가 100.00이 되어야 제출 가능
- 합계 불일치 시: 버튼 비활성 + 인라인 경고 ("합계가 100%가 되어야 합니다. 현재 N%")
- 투표 행사 후 수정 불가 (제출 전 확인 모달 제공)

**명수 계산 알고리즘:**

1. 각 선택지별 명수 계산: `위임 총 건수 × (% ÷ 100)`
2. 소수점 1자리에서 반올림하여 정수 명수 확정
   ```
   예) 위임 10건, 찬성 33.33% → 10 × 0.3333 = 3.333 → 3.3 → 3명
       위임 10건, 반대 33.33% → 3명
       위임 10건, 기권 33.34% → 10 × 0.3334 = 3.334 → 3.3 → 3명
       합계 9명 → 총 건수(10명)와 1명 차이 발생
   ```
3. 반올림 후 합계가 총 위임 건수와 맞지 않을 경우: **차이만큼 가장 높은 % 선택지에 자동 가감**
   ```
   위 예시: 33.34%인 기권이 최대 % → 기권 3→4명으로 자동 보정 → 합계 10명
   ```
4. 확인 모달에서 보정된 최종 명수 표시 후 관리자 최종 확인

**개별 표 랜덤 배정:**
- 각 선택지에 배정된 명수만큼, 의장 귀속 위임인 중 **누구의 표가 어느 선택지로 가는지 시스템이 무작위 배정**
- 특정 위임인의 표가 어느 선택지에 배정됐는지는 외부에 노출하지 않음 (감사 로그에만 기록)
- 목적: % 배분 의도 반영 + 개별 위임인의 투표 방향 비특정화

**확인 모달:**
```
"의장 귀속 위임 N건을 아래와 같이 행사하시겠습니까?
찬성 X명 / 반대 Y명 / 기권 Z명"
```

# 12. 보안 고려사항

## 12-1. 인증 및 접근 제어

| 항목 | 적용 방안 |
|---|---|
| 중복 투표 방지 | vote_logs 테이블에 member_number + vote_id + agenda_id UNIQUE 제약 |
| 이중 인증 | 이니시스 인증 + 회원번호 매칭 2단계 |
| Rate Limiting | 동일 IP 기준 분당 10회 이상 인증 요청 차단 |
| 예외 요청 어뷰징 방지 | 동일 투표 + 동일 이름 + 동일 회원번호 중복 요청 차단 |
| 세션 고정 공격 방지 | 로그인 후 세션 ID 재생성 (session_regenerate_id) |

## 12-2. 데이터 보호

| 항목 | 적용 방안 |
|---|---|
| 민감 정보 암호화 | 생년월일, 휴대폰번호 → AES-256 암호화 저장 |
| 비밀번호 해싱 | bcrypt (cost factor 12 이상) |
| HTTPS 강제 | HTTP → HTTPS 리다이렉트, HSTS 적용 |
| 쿠키 보안 | Secure + HttpOnly + SameSite=Strict |
| SQL Injection | PDO Prepared Statement 사용 |
| XSS 방지 | 출력 시 htmlspecialchars 처리, CSP 헤더 |
| CSRF 방지 | 폼 제출 시 CSRF 토큰 검증 |

## 12-3. 투표 무결성

| 항목 | 적용 방안 |
|---|---|
| 결과 조작 방지 | vote_logs 테이블 직접 수정 불가 (관리자 포함) |
| 감사 로그 | 모든 투표 행위, 관리자 행위 기록 (삭제 불가) |
| IP 기록 | 모든 인증 및 투표 시 IP 기록 |
| 이니시스 uniqueId | 동일 인증 ID로 중복 투표 방지 |
| 위임 무결성 | 위임 신청 후 본인 직접 투표 불가 처리 (DB 레벨) |
| 수임인 검증 최소 응답 | 이름+회원번호 일치 여부(bool)만 반환 — 회원 정보 조회 불가, 자동완성·검색 미제공 |
| 스냅샷 수정 감사 추적 | 관리자가 스냅샷 회원 정보를 수정할 경우 수정 전·후 값, 수정 사유, 수정자, 수정 일시를 감사 로그에 영구 기록 (삭제 불가) — A-15에서 조회 가능 |

## 12-4. 인프라 보안

| 항목 | 적용 방안 |
|---|---|
| 관리자 접근 제한 | `/manage/` 경로 IP 화이트리스트 고려 |

## 14-2. Fetch 기본 구조

```javascript
// assets/js/api.js
const API_BASE = '/api';

async function apiRequest(method, endpoint, body = null) {
  const options = {
    method,
    headers: {
      'Content-Type': 'application/json',
      'X-CSRF-Token': getCsrfToken(),
    },
    credentials: 'same-origin',
  };
  if (body) options.body = JSON.stringify(body);

  const res = await fetch(`${API_BASE}${endpoint}`, options);
  const json = await res.json();

  if (!res.ok || !json.success) {
    throw new Error(json.message || '오류가 발생했습니다.');
  }
  return json.data;
}
```

## 14-4. 인증 토큰 흐름

**투표자 세션:**
```
이니시스 인증 성공

  → 회원번호 입력 + 매칭 성공
  → 투표 세션 토큰 발급 (PHP $_SESSION)
  → 이후 API 요청마다 세션 쿠키 자동 첨부
```

**관리자 세션:**
```
ID/PW 로그인 성공
  → PHP $_SESSION 생성
  → 세션 쿠키 설정 (Secure, HttpOnly, SameSite=Strict)
  → 권한 레벨 세션에 포함
```

## Phase 4 — 권장 장기 개선 (정회원 로그인 시스템)

> **권장 배경:** 현재 구조에서 이니시스 간편인증은 접속할 때마다 비용이 발생한다. 투표 기간 중 위임 수정, 재확인 등 다중 접속이 발생하면 불필요한 인증 비용이 누적된다. 정회원 로그인 시스템을 도입하면 이니시스는 최초 1회 신원 확인 용도로만 사용하고, 이후 접속은 ID/PW 로그인으로 처리할 수 있다.

| 기능 | 설명 |
|---|---|
| 정회원 계정 시스템 | 이니시스로 신원 확인 후 ID/PW 계정 생성 — 이후 이니시스 재호출 없이 로그인 |
| 계정 연동 | 회원번호 기준으로 기존 투표 이력 유지 |
